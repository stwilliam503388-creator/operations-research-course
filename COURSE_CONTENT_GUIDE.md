# 课程内容质量规范

本规范用于统一 9 门课程的专业性、正确性、真实性和易懂性。它不要求每章都写成学术论文，而是要求每个核心概念和案例都做到：说人话，但不误导；能跑通，但不假装等于工业系统。

## 1. 核心写作原则

### Karpathy 四原则

修改课程内容前先遵守四条执行原则：

| 原则 | 在课程中的含义 | 验证方式 |
|------|----------------|----------|
| Think Before Coding | 先说明假设、取舍和不确定点，不把教学模型伪装成工业系统 | 每个案例能看到模型假设和现实边界 |
| Simplicity First | 用最小清晰改动提升理解，不为了显得专业堆定义或抽象 | 每段新增内容都能回答“读者会因此更懂什么” |
| Surgical Changes | 保留原有“人话教学”风格，只修正误导、补边界、加桥梁 | 不做无关重排，不统一重写所有章节 |
| Goal-Driven Execution | 每项优化都有可验证目标 | 能用验收标准、运行命令或自检问题验证 |

### 先让读者愿意读

深入浅出不是把内容写浅，而是安排理解顺序：

1. 先讲一个具体冲突：读者正在面对什么选择
2. 让读者先猜一个朴素做法：平均值、最近邻、贪心、只看成本
3. 说明朴素做法哪里会失败：用反例或边界场景打破直觉
4. 再引入模型：变量、约束、目标函数是为了解决哪个失败点
5. 回到业务：模型结果能指导什么，不能指导什么

每个重点案例建议至少有一个“停一下，想一想”问题和一个“如果你只记住一件事”总结。

### 人话解释要配严格边界

核心术语建议采用四层结构：

| 层次 | 作用 | 写法 |
|------|------|------|
| 一句话人话 | 先让读者抓住直觉 | 用类比、业务场景或反差表达 |
| 严格一点说 | 防止概念走样 | 给出必要条件、数学定义或适用范围 |
| 容易误解 | 主动拆掉错误直觉 | 写出“不是……”或“不能推出……” |
| 在本课程案例中怎么用 | 连接到代码和业务 | 指向变量、函数、图表或结果 |

示例：

> p 值不是“原假设为真的概率”。严格一点说，它是在原假设为真时，观察到当前结果或更极端结果的概率。它能说明“这个结果在原假设下罕不罕见”，不能直接说明“原假设有多大概率是真的”。

### 教学模型要明说假设

每个案例至少说明：

- 我们假设了什么
- 为什么这些假设适合教学
- 现实中什么时候会失效
- 失效后应该考虑什么扩展模型

### 不把最优解包装成业务答案

求解器返回 `optimal` 只表示“在当前模型、当前数据、当前约束下数学最优”。它不自动保证：

- 数据真实
- 假设合理
- 业务可执行
- 对未来仍然稳健
- 决策者一定接受

## 2. 案例标准结构

新增或重写案例时，优先采用以下结构：

1. 故事开场：读者在现实中遇到了什么选择
2. 直觉猜法：一个聪明但不完整的朴素方案
3. 为什么直觉会错：用反例或边界场景说明问题
4. 模型假设：哪些现实因素被保留，哪些被简化
5. 数学模型：变量、目标、约束
6. 方法选择：为什么用这个方法，不用别的方法
7. 业务-公式-代码映射：业务对象如何落到符号、Python 变量、C++ 变量
8. Python 实现：主线版本，强调可读性和可视化
9. C++ 对照：只在有教学价值时加入，强调算法细节、性能或工程接口
10. 运行结果解读：结果说明了什么，不能说明什么
11. 常见误区：列出 2-3 个最容易错的直觉
12. 停一下，想一想：让读者主动预测一个参数变化或异常结果
13. 如果你只记住一件事：用一句话留下核心认知
14. 现实局限与工业扩展：真实项目还需要补什么
15. 学习验收标准：读者学完后应能解释、修改、运行什么

## 3. 业务-公式-代码映射表

案例中出现数学模型时，建议补一张映射表：

| 业务对象 | 数学符号 | Python 变量 | C++ 变量 | 业务含义 |
|----------|----------|-------------|----------|----------|
| 工厂产能 | `S_i` | `supply[i]` | `supply[i]` | 工厂最多能发出的货量 |
| 单位运费 | `c_ij` | `cost[i][j]` | `cost[i][j]` | 从工厂到仓库的单位成本 |

如果某个案例没有 C++ 对照，`C++ 变量` 可以写“暂无”。

## 4. 高风险概念表达规范

### 概率与统计

- 置信区间：不要写“真值有 95% 概率落在这个区间”。应写“用同样方法反复抽样构造区间，约 95% 的区间会覆盖真值”。
- p 值：不要写“原假设为真的概率”。应写“在原假设为真时，观察到当前或更极端结果的概率”。
- 相关性：不要暗示相关就是因果。

### 运筹优化与 MIP

- LP 解天然整数只适用于特定结构，例如纯运输问题的全酉模矩阵，不要推广到所有带整数含义的问题。
- `optimal` 是模型内最优，不等于业务上可执行或长期最优。
- Big-M 是建模工具，不是越大越安全；过大的 M 会削弱 LP 松弛并拖慢求解。

### 随机规划与鲁棒优化

- 随机规划依赖场景和概率质量，场景质量差时结果也会差。
- 鲁棒优化不是越保守越好，保守度太高会牺牲期望收益和可执行性。
- 分布鲁棒优化不是“不需要数据”，而是在有限数据下保护分布估计误差。

### 多目标优化

- 非支配解是候选方案，不是最终推荐方案。
- 帕累托前沿帮助展示权衡，最终选择仍需要偏好、约束或决策者参与。
- 加权求和法简单，但可能漏掉非凸前沿上的 Pareto 解。

## 5. C++ 案例定位

C++ 案例不要求机械翻译 Python。优先选择以下场景：

- 算法细节：排序、图算法、动态规划、非支配排序
- 性能差异：大规模循环、场景评估、仿真
- 工程接口：文件读写、求解器 API、模块封装

默认约束：

- 使用 C++17
- 优先标准库
- 单文件可用 `g++ -std=c++17 -O2` 编译
- 只有课程主题确实需要时才引入第三方库或 CMake

## 6. 学习验收标准模板

每门课和每个重点案例建议给出验收标准：

- 能用自己的话解释 3-5 个核心概念
- 能指出模型的 2-3 个关键假设
- 能运行 Python 案例并解释主要输出
- 能运行 C++ 对照案例（如有）并说明它强调什么
- 能修改一个参数并预测结果变化方向
- 能说出这个教学模型在真实业务中还缺什么

## 7. 去枯燥化检查清单

如果一节内容连续出现定义、公式、表格，至少补一个读者友好元素：

| 枯燥信号 | 推荐补法 |
|----------|----------|
| 连续 3 个以上术语 | 加“小白版 / 专业版”双层解释 |
| 连续公式推导 | 每行公式后补一句业务含义 |
| 代码和公式割裂 | 加“这段代码对应模型哪一行” |
| 结果表格很多 | 加“读这张表先看哪一列” |
| 读者容易误信直觉 | 加“错误直觉 → 为什么合理 → 反例 → 正确理解” |
| 章节结尾突然结束 | 加“如果你只记住一件事” |

互动问题可以很短，但要能促使读者预测：

- 如果需求翻倍，最优解一定翻倍吗？
- 如果求解器说 infeasible，第一件事该查什么？
- 如果仿真跑 100 次和 10000 次结果不同，该信哪个？
- 如果成本权重调高，前沿上的方案会往哪里移动？

避免为了轻松而牺牲严谨：

- 不用夸张商业承诺代替验证
- 不用段子代替定义
- 不把“教学版能跑”说成“工业版可用”
- 不把复杂概念说成没有代价
