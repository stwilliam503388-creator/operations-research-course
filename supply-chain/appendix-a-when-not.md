# 附录A：什么时候不需要库存模型

不是所有库存问题都需要用数学模型解决。事实上，有时候建模反而浪费时间。以下是几个「先别建模，直接动手」的场景。

| 场景 | 理由 |
|------|------|
| 你的库存决策只有 2-3 个选项 | 穷举比建模快。比如「订 A 品牌的货还是 B 品牌的货」，直接算总成本就行 |
| 需求完全不可预测（连分布都不知道） | 先做需求预测，库存模型建立在预测之上。没有预测，任何模型都是空中楼阁 |
| 供应链极度不稳定（每周换供应商） | 优化一个短期会变的东西没意义。先把供应链稳定下来，再谈优化 |
| 库存占运营成本不到 1% | 不值得花精力优化。把精力放在更大的成本项上 |
| 业务方只接受「凭经验」 | 库存模型的解释成本可能高于收益。有时候一个简单规则比一个精确模型更容易被接受和执行 |

---

## 详细场景分析

### 场景 1：决策空间小

如果你的问题只有 2-3 种可能方案（比如「用 FedEx 还是顺丰」），手动计算总成本可能只需要 5 分钟。建立一个完整的 EOQ 模型反而需要 30 分钟输入数据、验证假设、解释结果。**原则：如果穷举比建模快，就不要建模。**

### 场景 2：数据不可用

库存模型对输入数据敏感——没有可靠的需求数据、提前期数据、成本数据，模型的输出就是数字垃圾。更糟糕的是，管理层可能会把垃圾数字当成科学决策的依据。**原则：数据质量比模型复杂度重要 10 倍。**

### 场景 3：业务环境高度动荡

如果供应商每个月都在换、或市场需求受政策影响大起大落，历史数据对未来的指导意义很弱。此时更值得做的是建立快速响应的供应链能力（缩短提前期、柔性产能），而不是优化一个注定过时的库存基线。**原则：在动荡环境中，速度比精度重要。**

### 场景 4：库存成本占比小

如果你的库存成本只占总运营成本的 0.5%，把精力放在优化库存上不如去优化物流、采购或生产。先抓大放小，等大项优化完了再回来处理小项。**原则：80/20 法则——先把 80% 的精力放在 80% 的成本上。**

### 场景 5：组织不接受

这是最容易被技术人员忽略的场景。你花了两周建了一个漂亮的模型，输出了一份详尽的分析报告——但业务方说「我们还是用老办法吧」。原因可能不是模型不够好，而是：对方不理解模型假设、不信任黑箱输出、或者觉得模型剥夺了他们的决策权。**原则：模型接受度 = 模型质量 × 沟通效果。前者是工程师的事，后者同样是。**

## 什么时候应该建模

反过来讲，以下场景建模的收益最大：

| 场景 | 建模收益 |
|------|---------|
| 高频重复决策（每周/每天做补货决策） | 每次决策省 5 分钟，一年就是几百倍回报 |
| 多 SKU 管理（几百上千个 SKU） | 人工逐 SKU 决策不可行，必须自动化 |
| 供应链协同（多个节点联动） | 没有模型就看不清全局效应 |
| 成本敏感性分析 | 模型可以快速回答「如果提前期增加 5 天会怎样」 |
| 场景对比（建不建新仓库？） | 模型可以量化不同方案的成本差异 |

---

## 一页纸决策指南

```
你的问题适合建模吗？
├─ 决策空间 ≤ 3 个选项？       → 穷举，不建模
├─ 没有可靠数据？              → 先搞数据，不建模
├─ 供应链正在剧烈变动？         → 先稳定，不建模
├─ 库存成本占比 < 1%？         → 不建模，做别的
├─ 组织不接受数学模型？         → 先科普，不建模
└─ 以上都不是 → 建模！选对应的库存模型
```

**一句话总结**：库存模型适用于「重复性高、有数据支撑、优化收益大」的场景。一次性、不可预测、成本占比低、组织不配合的情况——别建模，直接干。
