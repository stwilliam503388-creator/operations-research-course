# 低复杂度算法设计与优化技术基础课程

> 一个周末，从零到能识别问题复杂度、设计低复杂度算法解决实际问题。

## 这份教程是什么

这是一份给**有编程基础（能写 Python/C++）、对数据结构有基本了解（数组、链表、树）、但算法复杂度分析停留在「背 Big-O」阶段**的技术人写的自学教程。它不讲虚的——每一章都围绕「为什么这个算法快/慢」和「这个问题到底长得像哪种经典模式」展开。读完以后，你不会成为 ACM 金牌选手，但你一定会：

- 拿到一个新问题，10 分钟内判断出「这题的复杂度下限在哪」
- 把「暴力解」系统地改进到「低复杂度解」
- 看懂 LeetCode 竞赛题解区说的「DP + 状态压缩」、「分治优化」、「单调队列加速」是什么意思，并且自己也能写出来

## 术语速查表

下面的表是你翻任何一章时会反复回来查的东西。每个术语只有一句话解释——够你「认出来」就行，不需要背。

| 中文 | English | 一句话人话 |
|------|---------|-----------|
| 时间复杂度 | Time Complexity | 输入规模翻倍后，你的程序慢多少倍——O(n) 是翻倍后慢一倍，O(n²) 是翻倍后慢四倍 |
| 空间复杂度 | Space Complexity | 你的程序用了多少额外内存——包括输入的、偷的、借的所有空间 |
| 大O记号 | Big-O Notation | 只关心增长趋势，不关心常数——就像只看增长率不看起跑线 |
| 分治 | Divide & Conquer | 看不懂的大问题切成几个看得懂的小问题，分别解完再合并 |
| 动态规划 | Dynamic Programming | 把问题的答案记在小本本上，下次遇到同样的问题直接抄答案，不用重算 |
| 贪心 | Greedy | 每一步选当前最好的，赌全局也是最好——短视但有时候是好事 |
| 状态空间 | State Space | 一个问题所有可能的「情况」——你站在每个状态时面临的选择 |
| 状态转移方程 | State Transition | 从一个状态到下一个状态的「规则说明书」 |
| 最优子结构 | Optimal Substructure | 问题的最优解里藏着子问题的最优解——先做对每一小步，最后才能走对整条路 |
| 无后效性 | No Aftereffect | 过去已经发生了的事，不会改变你未来的决策逻辑——翻篇就是翻篇 |
| 记忆化 | Memoization | 把算过的结果存起来，下次要查不重算——跟缓存一个道理 |
| 递推 | Recurrence | 从最小的问题开始往上推——你知道了第1天和第2天，就能推出第3天 |
| 递归 | Recursion | 函数调自己——把大问题拆到不能再拆为止，再一层层返回答案 |
| 剪枝 | Pruning | 提前发现「这条路走的再多也不可能是最好的」，掉头 |
| 状态压缩 | State Compression | 用整数（bitmask）来代表一个「已选了哪些元素」的状态——节省空间，方便位运算 |
| 滑动窗口 | Sliding Window | 用一个窗口扫过数组，窗口内维护你需要的统计信息——O(n) 的典型代表 |
| 双指针 | Two Pointers | 两个指针对着一维数组从头尾或同向移动，配合缩小搜索范围 |
| 单调栈/队列 | Monotonic Stack/Queue | 栈或队列里的元素永远保持递增/递减——帮你快速找到下一个更大/更小的元素 |
| 回溯 | Backtracking | 试错——走不通就退一步换一条路。穷举但不傻举 |
| 二分答案 | Binary Search on Answer | 不知道答案具体是多少，但是猜一个值然后问「比这个值可行吗？」——像猜数字游戏 |
| 摊还分析 | Amortized Analysis | 单次操作偶尔很慢但平均下来很快——就像每年双十一抢购很堵，但平均到全年还是很通畅 |
| NP完全 | NP-Complete | 计算机科学家还没找到快速解法的问题——如果你能证明你的问题跟它等价，就别追求完美解了 |

> 读到任何一个术语忘了什么意思，回到这里看「一句话人话」列，3 秒搞定。

# 第一部分：复杂度分析与算法设计基础

> 在这一章，你不会看到一个「最优解」的代码，但你会看到：为什么 O(n) 和 O(n²) 的区别，在日常工作中意味着「秒出结果」vs「去喝杯咖啡」？一个问题怎么被拆成「不可再拆的原子操作」？以及——当你面对一个一分钟还没跑完的程序，从哪开始分析？

---

## 1.1 没有复杂度分析之前，世界是什么样子的

先想象一个场景。

你写了一段代码处理客户订单数据。测试环境跑 100 条订单，0.3 秒就出结果了——漂亮。上线以后，生产环境每天有 10 万条订单——结果跑了 8 个小时还没出完。老板站在你身后问：「怎么回事？」

你的第一反应可能是「服务器不够快」「数据库太慢了」「网络延迟」。但大部分情况下，真正的问题不是硬件，而是**算法选错了**。

### 增长率的直觉：为什么常数不重要

用一张图来建立直觉：

```
运行时间 vs 输入规模增长

T(n)
│
│                  O(n²)  ──▶ 每翻一倍，时间 x4
│                      ╱
│                   ╱
│                ╱        O(n log n) ──▶ 每翻一倍，时间略多于 x2
│             ╱           ╱
│          ╱          ╱
│       ╱         ╱            O(n) ──▶ 翻一倍，时间 x2
│    ╱        ╱              ╱
│ ╱       ╱               ╱
│──────────────────────────────────▶ n (输入规模)
```

**生活类比**：你在超市排队结账。

- **O(1)**：无论前面排了多少人，你直接走 VIP 通道——时间固定。
- **O(log n)**：你在一个巨大的图书馆里找一本书——每次你都能扔掉一半的书架，5 次后只剩一本。32 万本书 vs 64 万本书：只多查 1 次。
- **O(n)**：你在一个班里点名——人越多，点名越慢，成正比。
- **O(n log n)**：你整理一副扑克牌——牌越多要花的功夫略多于翻倍（因为还要比较大小）。
- **O(n²)**：一个舞会上每个人都要跟每个人握手——10 个人握 45 次，100 个人握 4,950 次，1000 个人… 已经没法玩了。

**工作类比**：你的程序处理 1,000 条记录 0.1 秒。客户说有 100 万条数据需要处理。

- 如果算法是 O(n)：0.1 × (1,000,000 ÷ 1,000) = 100 秒 ≈ 1.7 分钟。可以接受。
- 如果算法是 O(n²)：0.1 × (1,000,000 ÷ 1,000)² = 10,000 秒 ≈ 2.8 小时。应该在吃午饭时跑完。
- 如果算法是 O(n³)：已经 116 天了——不可能。

> 关键洞察：**99% 的性能问题不是「还不够优化」，而是「选错了算法」。** 在选算法之前谈微优化，是本末倒置。

## 1.2 一段八卦：算法复杂度理论是怎么从一个数学游戏变成软件工程必修课的

你不需要记住每个年份和名字。但你需要知道一件事：复杂度理论不是学术院的玩具——每一次重大的算法突破，都对应着某个工业场景的「天堑变通途」。

### 1965：大 O 记号的诞生

**故事主角**：Donald Knuth（高德纳）。

Knuth 在写《计算机程序设计艺术》（The Art of Computer Programming）第一卷时，需要一种方式来精确描述算法的「增长趋势」。前人（如 Paul Bachmann 在 1894 年）就提出过大 O 记号，但 Knuth 把它带进了计算机科学的核心——他在书中用它分析了每一种已知的排序和搜索算法。

生活类比：大 O 记号发明之前，算法分析就像厨师说「我炒这个菜需要一些盐」——每个人「一些」的意思都不一样。大 O 出来后，变成了「这道菜精确需要 2.3 克盐」。

### 1971：NP 完备性——找到了「最难」的问题

**故事主角**：Stephen Cook。

Cook 证明了：在计算复杂性理论中，存在一类问题——如果你能快速解决其中一个，你就能快速解决所有这一类问题。这就是 NP 完备性理论。这被后世称为「一个告诉你什么算不了的理论」。

**结果**：工程师们得到了一个极其实用的工具——如果一个新问题被证明是 NP 完备的，你就不用浪费时间寻找多项式解法了，直接上近似算法或启发式。

生活类比：你发现一个箱子的锁芯长得很奇怪。你找遍了所有钥匙都打不开，然后一个锁匠告诉你：「这种锁芯是德国某公司专利，没有万能钥匙——要么找专门的工匠开，要么砸锁。」这就是 Cook 定理给你的自由——不用再浪费时间找钥匙了。

### 为什么这段历史重要

**因为「复杂度」不是一个学术评价指标——它是一个部署决策工具。**

一家公司要在 10 万用户的系统上新增一个功能。两个工程师分别给出两种实现：

- 方案 A：代码简单（4 小时写完），但 O(n²)
- 方案 B：代码复杂（3 天写完），但 O(n log n)

没有复杂度分析，你只能选 A（因为看起来便宜）。有了复杂度分析，你知道：如果这个功能每天被调用 100 万次，方案 A 每年多浪费的算力 = 几万美元。**3 天的开发投入 vs 几万/年的运维成本——答案瞬间清晰。**

## 1.3 数学工具箱：你需要的不是证明，是直觉

### 对数：所有「快」的算法背后都是它

对数（log）的理解，是区分「能理解算法为什么快」和「不能理解」的分水岭。

**生活类比**：你在 1,024 页的电话簿里找一个人「王小明」。

- 办法 1（线性查找）：从第 1 页看到第 1,024 页。平均翻 512 页。
- 办法 2（二分查找）：翻到中间——「王」在左边还是右边？扔掉一半，重复。最多翻 **10 次**（因为 2¹⁰ = 1,024）。

翻倍规模到 1,048,576 页：办法 1 要翻 50 万页，办法 2 只需要翻 **20 次**。

**这就是 log₂ n 的魔力**：每一次操作都将问题规模减半。不依赖巧合，不依赖运气——只是数学。

| n | log₂ n | 差别说明 |
|---|--------|---------|
| 1,024 | 10 | 线性 1K 次 vs 二分 10 次 |
| 1,048,576 | 20 | 线性 1M 次 vs 二分 20 次 |
| 1,000,000,000 | ~30 | 线性 10亿次 vs 二分 30 次 |

**工作类比**：你的数据库有 100 万条记录。全表扫描 = 100 万次比较。加 B-tree 索引后 = 约 20 次比较。索引之所以快，底层的数学就是对数。

### 递归树：画出来，复杂度就写出来了

这是分析递归算法复杂度的最简单方法，比套主定理（Master Theorem）直观得多。

**生活类比**：你让一个朋友帮你传话，他让另一个朋友传话，最后一个朋友传到了—这就是递归的「拆」。当最后一个朋友把结果传回来，每一层的人都要把结果往上递—这就是「合」。

**画递归树的三步法**：

以归并排序为例，输入规模 n = 8：

```
层 0 (规模 n):     [8 个元素]        ← 1 个问题, 耗时 cn
                  /          \
层 1 (规模 n/2): [4]        [4]      ← 2 个问题, 各耗时 cn/2, 总 c·2·(n/2) = cn
                /   \      /   \
层 2 (规模 n/4): [2] [2]  [2] [2]    ← 4 个问题, 各耗时 cn/4, 总 c·4·(n/4) = cn
               / \  / \   / \  / \
层 3 (规模 1): [1][1][1][1][1][1][1][1] ← 8 个问题, 各耗时 c·1, 总 c·8·1 = cn
```

一共有 log₂ n + 1 层（这里是 4 层），每层总耗时 = cn。所以总复杂度 = cn × (log₂ n + 1) = **O(n log n)**。

**无论递归长什么样，画三层就知道趋势了**。

### Big-O 速查牌：10 个你永远会遇到的复杂度

| 记号 | 名称 | 当你看到它时 | 典型算法 |
|------|------|-------------|---------|
| O(1) | 常数时间 | 一次操作搞定 | 哈希表查找、数组随机访问 |
| O(log n) | 对数时间 | 每次砍掉一半 | 二分查找、平衡树操作 |
| O(n) | 线性时间 | 每个元素看一眼 | 线性扫描、遍历链表 |
| O(n log n) | 线性对数时间 | 「先拆再合」 | 归并排序、堆排序、快速排序平均情况 |
| O(n²) | 平方时间 | 嵌套循环 | 冒泡排序、插入排序、双层 for |
| O(n³) | 立方时间 | 三层嵌套 | 弗洛伊德全源最短路、朴素矩阵乘法 |
| O(2ⁿ) | 指数时间 | 所有子集 | 子集枚举、暴力 TSP |
| O(n!) | 阶乘时间 | 所有排列 | 全排列枚举、暴力分配问题 |
| O(n·2ⁿ) | 指数 × 线性 | DP 状态压缩 | 旅行商 DP (Held-Karp) |
| O(√n) | 平方根时间 | 筛选跨度减半 | 素数判定 (暴力) |

> **对于 1 秒内能处理的数据规模（现代 CPU ≈ 10⁸ 次操作）：**
> O(log n) → 宇宙大小都能处理；O(n) → 1 亿；O(n log n) → 1,000 万；O(n²) → 1 万；O(2ⁿ) → 30。

## 1.4 五大算法设计范式：上帝视角看大多数算法

下面的五个范式不是让你背的。它们是「模板」——当你面对一个新问题时，你大概率会发现它可以归结为这五个中的某一个（或某几个的组合）。每种范式都用一个统一格式讲：直觉 → 判断准则 → 模板代码骨架 → 典型问题。

### 范式一：分治 (Divide & Conquer)

**难度：★★☆☆☆**

**直觉**：大问题拆成更小的同类型问题，分别解决，然后合并结果。

**判断准则（何时用）**：
- 输入可以自然拆分成互不重叠的子集（数组可以切半、矩阵可以分块）
- 子问题的解能「合并」成原问题的解

**典型结构**：
```
def solve(problem):
    if is_base_case(problem):       # 原子问题，直接解
        return base_answer(problem)
    subproblems = divide(problem)   # 拆
    sub_answers = [solve(sub) for sub in subproblems]  # 递归解
    return combine(sub_answers)     # 合
```

**典型问题**：归并排序、快速排序、最大子数组和、Strassen 矩阵乘法、最近点对

---

### 范式二：动态规划 (Dynamic Programming)

**难度：★★★☆☆**

**直觉**：一个问题的最优解依赖于它的子问题的最优解。不重算已经算过的子问题。

**判断准则（何时用）——两条必须同时满足**：
1. **最优子结构**：问题的最优解包含子问题的最优解。你不能「全盘最优但局部次优」。
2. **重叠子问题**：不同的「路径」会到达相同的子问题状态。如果子问题不重叠，用分治就够了。

**五步法**：
```
Step 1: 定义状态 —— dp[i] 或 dp[i][j] 代表什么？（如「前 i 个物品、容量 j 时的最大价值」）
Step 2: 找转移方程 —— 从 dp[i-1] 如何到 dp[i]？
Step 3: 初始化 —— 最小子问题的答案（边界条件）
Step 4: 确定遍历顺序 —— 从最小到最大，还是从大到小？
Step 5: 返回结果 —— dp[n] 或 max(dp[...])
```

**状态设计口诀**：「一维用次数，二维加约束。三维以上想清楚了再用。」

**典型问题**：0/1 背包、最长公共子序列（LCS）、编辑距离、区间 DP（矩阵链乘）、树形 DP

---

### 范式三：贪心 (Greedy)

**难度：★★☆☆☆**

**直觉**：每一步都选当前最优的选择，而且永远不后悔。

**判断准则**：需要满足**贪心选择性质**——局部最优选择最终会导致全局最优解。这是最难证明的一条，但也是判断「能不能用贪心」的唯一标准。

**贪心 vs DP 的选择表**：

| 场景 | 该用贪心？ | 该用 DP？ |
|------|-----------|-----------|
| 找零钱（标准货币面额 1,5,10,25） | ✅ 关键：大面额整除小面额 | ❌ |
| 找零钱（特殊面额 1,3,4） | ❌ 会出错（3+3 比 4+1+1 更好） | ✅ |
| 活动安排（最多不相交活动） | ✅ 最早结束时间优先 | ❌ |
| 背包（可以切分物品） | ✅ 按单位价值排序 | ❌ |
| 0/1 背包（不能切分） | ❌ 局部最优不是全局最优 | ✅ |
| 最小生成树 | ✅ Prim/Kruskal | ❌ |
| 最短路径（无负权） | ✅ Dijkstra | ❌ |
| 最短路径（有负权） | ❌ | ✅ Bellman-Ford |

**典型问题**：区间调度、哈夫曼编码、最小生成树、Dijkstra 最短路径（无负权）

---

### 范式四：搜索 + 剪枝 (Backtracking & Pruning)

**难度：★★★☆☆**

**直觉**：暴力穷举所有可能，但一旦发现某条路不可能通向最优解，立刻掉头。

**判断准则**：
- 问题的解空间是有限的（可以通过搜索枚举）
- 有明显的**剪枝条件**——能在深度优先遍历早期就排除大量分支（如当前累计值已超过最优解）

**模板骨架**：
```
def backtrack(path, state):
    if is_solution(state):        # 找到一个可行解
        update_best(path)
        return
    if not feasible(state):       # 剪枝——这条路走不通了
        return
    for choice in choices(state): # 对当前状态的所有可能选择
        make_choice(choice)       # 做选择
        backtrack(path, state)    # 递归深入
        undo_choice(choice)       # 撤销选择（回溯核心）
```

**典型问题**：N 皇后、数独求解、全排列/组合生成、子集和

---

### 范式五：二分答案 + 判定 (Binary Search on Answer)

**难度：★★☆☆☆（概念简单，但不容易想到）**

**直觉**：直接求解复杂，但「猜一个数然后判断它是否可行」简单。那就猜——用二分法猜。

**判断准则（规律性极强，一学就会）**：
- 问题问的答案是**一个数值**（最小化最大值、最大化最小值、能否在 X 内完成）
- 这个数值有**单调性**——如果答案 X 可行，那么所有比 X「更宽松」的答案也一定可行

**模板骨架（记住这 6 行）：**
```
def can_do(X):      # 判断「X 作为答案是否可行」
    return ...      # 具体判定逻辑

lo, hi = 0, 1e9     # 答案的范围（下界和上界）
while lo < hi:      # 二分查找最小的可行值
    mid = (lo + hi) // 2
    if can_do(mid):
        hi = mid    # 可行，尝试更小的
    else:
        lo = mid + 1 # 不可行，尝试更大的
return lo
```

**典型问题**：木材切割（最长能切多长）、书籍抄写（最少时间）、修路问题、分巧克力

## 1.5 解题心法：拿到新问题，十分钟定位到正确的范式

这一节教的是：**你面对一道全新的算法题，从哪开始。** 不是刷题的数量，是思考的顺序。

### 第一步：看输入规模（30 秒，最重要）

这是唯一一个**客观**的信息，也是最快帮你排除/确认某些算法的依据：

| 输入规模 n | 暗示的复杂度上限 | 可以使用的算法范式 |
|-----------|----------------|-------------------|
| n ≤ 20 | O(2ⁿ) 或 O(n!) | 状态压缩 DP、暴力回溯、搜索+剪枝 |
| n ≤ 100 | O(n³) | Floyd、三重循环 DP |
| n ≤ 1,000 | O(n²) | 双层循环 DP、朴素 DP |
| n ≤ 10⁵ | O(n log n) | 排序、二分、线段树、分治 |
| n ≤ 10⁶ | O(n) | 滑动窗口、双指针、线性扫描、哈希表 |
| n ≤ 10⁸ | O(log n) 或 O(1) | 数学公式、二分答案、快速幂 |

> 例如：如果题目说 `1 ≤ n ≤ 10⁵`，你就不应该写 O(n²) 的解法——面试官直接打「需要优化」。

### 第二步：看问的问题（30 秒）

| 问法 | 猜测的范式 |
|------|-----------|
| 「最大/最小/最长/最短」 | DP 或贪心或二分答案 |
| 「有多少种方式」 | DP（计数型）或回溯 |
| 「是否存在」 | 二分判定或 DFS/BFS |
| 「是否存在且要求最优」 | 二分答案 + 贪心判定 |
| 「所有排列/组合」 | 回溯 |
| 「在 O(log n) 内」 | 二分查找、二分答案 |
| 「在 O(n) 内，不能用排序」 | 滑动窗口、双指针、哈希 |

### 第三步：尝试暴力解（2 分钟）

不要一开始就想最优解。先想：**如果我只能写最笨的方法，怎么写？** 这能帮你：

1. 确认你正确理解了问题
2. 确认输入规模——如果暴力解本身就在复杂度限制内（比如 n ≤ 20），你甚至不需要优化
3. 从暴力解的「冗余计算」中找到优化的切入点

### 第四步：问自己三个问题（1 分钟）

1. **「这个问题的子问题是否重叠？」** → 是 → 考虑 DP
2. **「假设我每一步选最好的——结果对吗？」** → 对 → 考虑贪心；不对 → 不能贪心
3. **「我能猜一个答案，然后快速验证它吗？」** → 能 → 考虑二分答案

### 第五步：画状态图（2 分钟）

用纸和笔画出小规模例子（n=3 到 n=5）的完整状态转移图。标出：
- 每个状态是哪些参数决定的（这就是 DP 状态定义）
- 从一个状态到另一个状态的「跳转成本」（这就是转移方程）
- 哪些参数是不变的（可以省略进状态定义）

**检验**：画完图后，手动执行一次转移，确保值正确。

### 五步法速查卡

```
Step 1 (30s)  → 看 n：n ≤ 20 暴力，n ≤ 10⁵ 考虑 O(n log n)
Step 2 (30s)  → 看问法：「多少种」= DP/回溯，「最值」= DP/贪心/二分
Step 3 (2min) → 写暴力解，确认理解正确
Step 4 (1min) → 三问：子问题重叠？局部=全局？能猜答案+验证？
Step 5 (2min) → 画小规模状态图，手算验证
```

> **逃生通道**：五步法中最容易被跳过的是第一步——但它是唯一客观的输入。跳过第一步直接想算法，你可能会写出一个 O(n²) 的解而实际需要 O(n log n)。先看 n。一切从 n 开始。

## 2.1 一张图看懂算法设计范式的协作关系

       ┌──────────┐     ┌──────────┐     ┌──────────────┐

## 2.2 分治的深度剖析

### 分治 vs 动态规划：分界线在哪？

这是初学者最容易混淆的一对。我们用一张表说清楚：

| 维度 | 分治 | 动态规划 |
|------|------|---------|
| 子问题关系 | 互不重叠 | 重叠（不同路径到达相同状态）|
| 典型结构 | 二叉树——从上往下拆，从下往上合 | 有向无环图——状态之间有转移 |
| 重复计算 | 无（子问题独立） | 有（需要记忆化） |
| 能否用递归 | 天然适合 | 可以，但容易爆栈，推荐递推 |
| 典型复杂度 | O(n log n) | O(n·state) 通常更大 |

**什么时候分治自然演变成 DP？** 当你发现合并子问题时，需要用到同一个子问题的很多副本——这就是重叠子问题的信号，可以考虑 DP 了。

### 分治三连问

| 问题 | 答案 |
|------|------|
| 能不能拆？ | 输入是否能切割为互不相交的子集？ |
| 能不能合？ | 子问题的解能不能在 O(合并复杂度) 内生成原问题解？ |
| 能不能更快？ | 如果拆分成 k 份，合并复杂度是 O(k) 还是 O(k²)？ |

---

## 2.3 DP 的核心：状态设计是艺术

### 状态设计的三个层次

```
Level 1: 一维 DP → dp[i]
  语义：前 i 个元素 / 以 i 结尾的⋯
  典型题：最大子序和、最长递增子序列、爬楼梯
  复杂度：O(n) 或 O(n²)

Level 2: 二维 DP → dp[i][j]
  语义：前 i 个元素在「第二维约束 j」下的答案
  典型题：0/1 背包、LCS、编辑距离
  复杂度：O(n·m) 或 O(n·W)

Level 3: 状态压缩 DP → dp[mask]
  语义：mask 的二进制为 1 的位表示「已选」
  典型题：TSP DP、集合划分
  复杂度：O(n·2ⁿ) — n ≤ 20 时可用
```

### 状态设计的「五分钟法则」

如果你花超过 5 分钟还没想出状态定义，换一条路：
1. 先假设你不能用 DP——试试别的范式
2. 如果确定要用 DP——从**最小的输入例子**（n=1）手动推导到 n=3，看状态怎么变化
3. 画出来——看到规律的瞬间，状态定义往往自动浮现

### 常见 DP 类型速查

| DP 类型 | 状态定义模式 | 转移特点 | 典型复杂度 |
|---------|-------------|---------|-----------|
| 线性 DP | dp[i] 前 i 个 | i 从 0 到 n 递推 | O(n) ~ O(n²) |
| 背包 DP | dp[i][w] 前 i 个物品容量 w | 选/不选两种转移 | O(nW) |
| 区间 DP | dp[l][r] 区间 [l,r] | 枚举中间分割点 k | O(n³) |
| 树形 DP | dp[u][state] 节点 u 的状态 | 子节点向上合并 | O(n·state) |
| 计数 DP | dp[i] 种数 | 累加所有可能的转移 | O(n) ~ O(n²) |
| 数位 DP | dp[pos][tight][state] | 按位枚举 | O(log n × states) |

## 2.4 贪心：最难的不是写，是证明

### 贪心正确性的两个支柱

```
1. 贪心选择性质：局部最优选择最终一定是全局最优的一部分
   → 证明手段：交换论证（Exchange Argument）

2. 最优子结构：做出贪心选择后，剩下的子问题的最优解 = 原问题在剩余部分的最优解
   → 证明手段：反证法
```

### 交换论证模板

```
假设存在一个全局最优解 OPT。如果 OPT 的第一步选择 ≠ 贪心选择：
  Step 1: 交换 OPT 中的「第一步」和「贪心选择」，得到新解 OPT'
  Step 2: 证明 OPT' 不比 OPT 差（目标函数值相等或更好）
  Step 3: 因此 OPT' 也是最优解
  Step 4: 归纳——对所有后续步骤重复，贪心选择构造出最优解
```

### 什么时候做「贪心错了」的快速测试

构造一个 n=3 或 n=4 的小输入，手动执行贪心策略，再用暴力穷举验证结果是否一致。如果不一致——你的贪心是错的，换 DP。

## 2.5 回溯+剪枝：暴力打不过就加条件

### 什么情况下回溯比 DP 更好？

| 情况 | 该用回溯还是 DP |
|------|---------------|
| 只需要一个解（而非最优值） | 回溯（DFS 找到即停） |
| 所有解都要列举 | 回溯 |
| n 很小且 DP 状态空间太大 | 回溯+强剪枝 |
| 需要最优值 | DP |
| 状态维数 ≤ 3 | DP（通常更快） |

### 剪枝的三个层次

```
Level 1 (基础)：可行性剪枝
  当前 partial 已不可能满足约束 → return
  例：N 皇后中，当前位置已被攻击

Level 2 (进阶)：最优性剪枝（用于求最优解）
  当前 partial 已不可能优于当前 best 值 → return
  例：已用成本 > 当前最优成本

Level 3 (高阶)：对称性剪枝
  两个分支本质等价 → 只探索一个
  例：组合选择中，选 {1,2} 和 {2,1} 是同一个组合
```

## 2.6 二分答案：很多人想不到它

### 识别二分答案的三个特征（全部满足才用）

1. **求最值**：最小化最大值 / 最大化最小值 / 求满足条件的最大/最小值
2. **有单调性**：如果 X 可行，所有大于（或小于）X 的值也一定可行
3. **判定函数简单**：给定一个 X，能在 O(n) 或 O(n log n) 内验证它是否可行

### 二分答案 vs 二分查找

| | 二分查找 | 二分答案 |
|--|---------|---------|
| 搜索对象 | 数组中的值 | 答案空间 |
| 前提 | 数组已排序 | 答案具有单调性 |
| 判定 | 比较大小 | 运行一个判定函数 |
| 复杂度 | O(log n) | O(log R · f(n)) |

**R = 答案范围，f(n) = 单次判定的时间复杂度。**

## 2.7 进阶技术：滑动窗口与单调栈

### 滑动窗口（Sliding Window）

**直觉**：维护一个动态的「窗口」，在扫描数组时窗口内容不断变化，你实时维护窗口内的统计量。

**判断准则**：
- 处理的是**连续区间**（子数组、子串）
- 窗口的收缩条件与展开条件清晰

**模板骨架**：
```
left = 0
for right in range(n):       # right 是窗口右边界
    add(arr[right])          # 将右边界元素加入窗口
    while not_ok():          # 窗口不满足条件时收缩左边界
        remove(arr[left])
        left += 1
    update_answer()          # 此时窗口满足条件，更新答案
```

**典型问题**：长度最小的子数组（≥ target）、无重复字符的最长子串、字符串排列包含

### 单调栈（Monotonic Stack）

**直觉**：维护一个栈，栈内元素保持单调（递增或递减）。当你需要「快速找出数组中每个元素的下一个更大（或更小）元素」时——用单调栈。

**判断准则**（规律性极强）：
- 问题中有「下一个更大/更小」或类似的比较语义
- 扫描顺序决定关系

**典型问题**：每日温度（下一个更大）、接雨水、柱状图中最大矩形、股票价格跨度

## 2.8 算法复杂度速查表

| 算法 | 时间复杂度 | 空间复杂度 | 稳定排序？ |
|------|-----------|-----------|-----------|
| 冒泡排序 | O(n²) | O(1) | ✅ |
| 插入排序 | O(n²) | O(1) | ✅ |

## 附录A：什么时候不需要优化

| 情况 | 理由 |
|------|------|
| 一次性脚本，跑完即弃 | 不优化 |
| n 永远 ≤ 100 | O(n²) 完全足够 |
| 瓶颈在 I/O 不在 CPU | 优化算法毫无意义 |
| 分析显示当前已够快 | 别优化——时间花在正确的地方 |
| 代码可读性 > 性能 | 为下一次维护的人着想 |

## 附录B：TOP 5 必踩坑

1. **忽略空间复杂度**：DP 得出 O(n²) 的空间但 n=10⁵ → 内存爆炸。在设计 DP 时就考虑滚动数组。
2. **二分答案的边界条件**：`while lo < hi` vs `while lo <= hi`，选哪一个？记住"寻找最小的可行值"的模板。
3. **DP 初始化错误**：别忘记 dp[0] 或 dp[0][0]。最隐蔽的错误是 dp[0] 可以是什么都不选的情况（值为 0），但有些题目需要初始化为 +∞ 或 -∞。
4. **贪心的错误使用**：没有验证贪心选择性质就直接用。记住「构造 n=3 反例测试」。
5. **分治的合并步骤复杂度低估**：合并如果用了 O(n²)，分治就白费了。合并必须跟分治不是一个复杂度量级。

## 附录C：算法优化与 ML 的交叉点

- **动态规划 + 强化学习**：Bellman 方程是 DP 和 RL 的共同基础
- **贪心 + 推荐系统**：多臂老虎机（Multi-Armed Bandit）的 ε-贪心策略
- **二分答案 + 模型评估**：在假设检验中二分搜索最优阈值
- **滑动窗口 + 时间序列**：实时特征工程中的滑动窗口统计
- **近似算法 + 大规模 ML**：当精确解太慢时，用近似算法换速度
