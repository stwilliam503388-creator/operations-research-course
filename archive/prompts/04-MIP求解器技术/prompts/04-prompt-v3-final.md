## 核心写作约束（每一章都遵守）

### 1. 先直觉，后公式
**永远不要上来就甩公式。** 每个新概念的结构是：
> ① 先给一个生活中能摸到的类比（1-2句话）
> ② 再用白话解释数学含义（不允许堆砌符号，必须逐符号翻译成中文含义）
> ③ 最后给出正式数学表达式（此时读者已经有心理模型了）

反例（禁止）：直接写「设 x∈{0,1}ⁿ 为决策变量，min cᵀx s.t. Ax ≤ b」
正例（要求）：先说「你在超市购物，每个商品要么拿要么不拿，总共不能超过预算——这就是一个背包问题。用数学怎么描述呢？我们用一个 0/1 变量 xᵢ 表示第 i 件商品拿不拿…」

### 2. 每个抽象概念必须配「一句话人话翻译」+ 工作场景双类比
关键概念（LP松弛、对偶、分支定界、割平面、Gap）必须给**两个类比**：一个生活类比 + 一个工作类比。

例如：
- 「LP 松弛」→ 生活类比：「假装整数变量可以是小数，先把问题变简单算一下」→ 工作类比：「就像你做预算，先不算税费和零头，看大数对不对」
- 「对偶」→ 生活类比：「从卖家的角度重新看同一个交易——买家花的钱 = 卖家收的钱」→ 工作类比：「找外包还是自己干——如果你发现自己做一件事情的内部成本高于市场价，就该外包」
- 「割平面」→ 生活类比：「在不切掉任何合法答案的前提下，画一条线把最优但非法的分数解划出去」→ 工作类比：「筛简历时划掉学历不达标的——不切掉任何合格的候选人，只切掉明显不行的」
- 「分支定界」→ 生活类比：「猜谜游戏——每次猜一个变量是 0 还是 1，猜错了就回头重新猜」→ 工作类比：「排查 bug 时二分定位——先测前半段，如果不在这半段就直接跳过」
- 「Gap」→ 生活类比：「你现在离第一名还差百分之几——0% 就是你已经追平了" → 工作类比：「你跟竞争对手的价格差——Gap=5% 代表你贵 5%，不一定非要追到 0%」

### 3. 不堆术语，建立术语地图
- 每个专业术语首次出现时必须用括号标注英文全称

### 第一部分：背景知识 — 搞懂 MIP 到底在解决什么问题（约占 25%）

#### 1.1 没有优化之前，世界是什么样子的
- 开场用一个企业实际困境引入：「你管着 50 辆货车、300 个配送点，怎么安排路线最省钱？试试人工排——一天排出来只能跑完一半，剩下的一半客户炸了。」
- 从直觉到数学：决策、约束、目标——把这三个概念像「做饭三要素」一样讲清楚
  - 决策变量 = 你要决定什么（买什么菜、做几道）
  - 约束 = 什么不能做（冰箱装不下、只有两个小时）
  - 目标 = 你想达到什么（全家吃饱且不超预算）
- 连续 vs 离散的直观类比：「水量可以任意调节（连续），但人数只能是整数（离散）——你不能派 2.7 个工人去干活」
- LP → IP → MIP 的递进关系：用 Venn 图 ASCII 画出来
- 一句话总结：「MIP = 有些变量必须是整数的线性规划。这个'有些'让问题从简单变得极难，但也让它能描述几乎所有真实世界的决策问题。」

> 🆘 卡住了怎么办：其实你只需要记住三件事——决策变量、约束、目标。这就是全部的数学优化。先把这六个字刻在脑子里，后面所有的技术都是为了让计算机帮你找答案。

#### 1.2 一段八卦：MIP 是怎么从一个数学游戏变成千亿美元产业的
- 1947 Dantzig 的单纯形法——他在 RAND 公司做军事后勤时「顺便」发明了它。故事细节：他导师误以为一个未解问题只是课后作业，Dantzig 两周后交了上去
- 1954 旅行商问题——三个人证明了这个看起来简单的问题实际上是「NP-hard」级别的灾难
- 1960 Land-Doig 分支定界——首次给出求解整数规划的通用框架
- 1984 Karmarkar 内点法——AT&T 的数学家发现了一条完全不同的路
- 1990s 以后：CPLEX 商业化 → Gurobi（CPLEX 核心团队出走创业）→ SCIP（开源界的灯塔）→ COPT（中国团队突围）
- 关键时间线用 ASCII 时间轴：

```
1947 ─── 1960 ───── 1984 ──────── 1990s ─────── 2010s ─── 2020s
单纯形   分支定界   内点法       CPLEX商业化   Gurobi崛起  COPT国产
                               SCIP开源                  HiGHS新生代
```
- 为什么要讲这段历史？因为你会理解：MIP 不是一个纯数学玩具——它背后是真实的商业竞争和工程迭代。今天你能用几行 Python 调用 Gurobi 求解上百万变量的问题，是几代数学家+工程师+创业者接力几十年的成果。

> 🆘 这节没有技术考点，当成背景故事看就行。核心记住：MIP 能走到今天是因为工业界拼命砸钱——他们真的需要这个。

#### 1.3 数学工具箱：你需要的不是证明，是直觉
- 多面体与极点：「想象一个橡皮筋套在多面体上，它会崩到哪个角？那就是单纯形法在找的极点」
- 对偶理论：
  - 生活类比：「你开一家奶茶店，竞争对手也开一家。你的定价策略和他的竞价策略之间存在一个神奇的等式——你卖奶茶赚的钱的上限 = 他从原料商那里买原料的成本的下限」
  - 工作类比：「找外包还是自己干——对偶变量的值（影子价格）告诉你：如果一个资源的内部成本高于市场价，就该外包」
- NP-hardness：
  - 生活类比：「不是锁本身打不开——是全宇宙的计算机一起上也打不开。但换个锁匠（建模角度）或者换把锁（近似求解）可能有办法」
  - 工作类比：「MIP 的 NP-hard 意味着你没有万能解法。但它不意味着你放弃——你只是要学会挑选适合你问题结构的技术」
- 松弛：
  - 生活类比：「考试时先把大题跳过做小题，拿完基础分再回头攻大题」
  - 工作类比：「先去掉最难满足的条件看个大概，然后再看看能不能把条件加回去」

> 🆚 对偶是本章最难的概念。如果没看懂，先记住一句话：「对偶 = 同一个最优化的两种等价描述」。这在第 2 章会反复用到。

#### 1.5 建模心法：拿到新问题，五分钟写出第一个模型（新增）
这是初学者最需要的章节——不是「每个问题长什么样」，而是「面对空白屏幕从哪开始」。

**步骤一：画图（最重要，花 2 分钟）**
别写公式。在白板上画实体和它们之间的关系。标出输入、输出、中间环节。举例：做配送问题——画三个点代表仓库，十个点代表客户，连线代表可能的路线。图上画完你就知道变量该放哪里了。

**步骤二：找决策变量（一句话口诀）**
问自己：「最后输出给领导看的那张 Excel 表里，哪些单元格是需要填的？」那些要填的就是决策变量。比如配送表里的「哪辆车送哪一单」是变量，「每辆车的载重上限」是常量。

**步骤三：写约束清单（用自然语言，别急翻译）**
在白话里列出所有「不能」：
- 「不能超过车载重」
- 「每单只能被一个人送」
- 「总花费不能超过预算」
写完之后检查：有没有漏掉的物理限制？

**步骤四：写目标（通常一句话）**
「让 X 最小/最大」，其中 X 通常是钱、时间、距离、利润。注意别写反——最大化利润和最小化成本不是等价的。

**步骤五：检查边界**
- 带 0 个订单会怎样？（求解器应该返回 cost=0 和一个空方案）
- 带 1,000,000 个订单会怎样？（求解器应该报内存不足或超时——这是正常的，不是你的模型错了）
- 去掉整数约束跑 LP 松弛——得到的值有意义吗？（应该 ≤ 整数情况的最小值，如果不是，模型写反了）

> 🆘 如果连步骤一都画不出来——你还没理解业务。回去找业务方聊清楚，别急着建模。

```
│  │② LP Solver      │ │ ← 解 LP 松弛，提供下界
│  │③ Branching      │ │ ← 决定在哪个变量上分支
│  │④ Cutting Planes │ │ ← 切掉分数解区域
│  │⑤ Heuristics    │ │ ← 快速找可行解
│  │⑥ Node Selection │ │ ← 决定下一个看哪个节点
│  └─────────────────┘ │
└──────────────────────┘
       │
       ▼
输出: 最优解 + Gap + 求解日志
```

- 一句话介绍每个组件（像介绍篮球队六个人的位置）：
  1. Presolve = 赛前热身，简化问题
  2. LP Solver = 发动机，反复解松弛版本
  3. Branching = 分叉决策——关键变量猜 0 还是 1
  4. Cutting Planes = 削边界——把没用的搜索空间砍掉
  5. Heuristics = 狙击手——快速找到一个不差但未必最优的解
  6. Node Selection = 导航——在搜索树里决定下一步走哪

- 告诉读者：「你现在不需要理解每个组件，只需要知道它们在流水线上的大致位置和一个词概括就够了。接下来我们一个一个拆开看。」

> 🆘 如果你觉得这图太复杂——先把 ② LP Solver 和 ③ Branching 理解了就行。④⑤⑥ 是锦上添花。

#### 2.2 Presolve：模型还没开始算，就已经被「瘦身」了
- 类比：「你写代码时编译器做的优化——把没用的变量删了、常量表达式算出来——Presolve 就是求解器的编译器」
- 四种主要技术，每种配一个 3-4 行的极端简化例子：
  - **边界收紧**：「如果 x ≤ 5 且 x 在另一个约束里 ≥ 3，那 x 的范围直接锁定在 [3,5]」
  - **冗余删除**：「如果约束说 x ≤ 10，但其他约束已经锁死 x ≤ 5，那 x ≤ 10 这条可以直接删」
  - **系数化简**：「15x + 30y ≤ 120 两边同时除以 15 → x + 2y ≤ 8」
  - **变量替换**：「如果 z = x + y 且你可以直接用 z 替代，那 x 和 y 可以被移除」
- **关键冲击力数字**：一个典型工业模型 presolve 后可能从 100 万变量缩到 10 万。用同一个例子走一遍 presolve 前后对比。
- 注意事项：Presolve 几乎总是好事，但极端情况下可能改变问题的数值性质——如果你跑完发现结果诡异，可以试试把 Presolve 关掉 (`Presolve=0`) 对比。

> 🆘 这节不用记具体技术名。记住一句话：求解器比你以为的聪明得多——它会在你开始等之前就把模型自动瘦身。

#### 2.3 LP求解器：MIP的发动机
- 开头定调：「别被线性规划这个词吓到，它的核心问题是——在几条直线围成的区域里找一个点，让某个值最大。初中生画图都能做。」
- 单纯形法：用 「爬山」类比
  - 「你蒙着眼睛站在多面体（钻石形状）的表面上，目标是最低点。每次你摸到一个顶点，就往更低的相邻顶点走。走到没有更低的地方了，你就到了。这就是单纯形法。」
  - 为什么叫单纯形？Dantzig 最初的灵感来自一个三维多面体模型
- 内点法：用「钻隧道」类比
  - 「不沿着边缘走，而是从内部钻一条弯曲的隧道直通最优解。对大问题比爬山更快。」
- 一张对比表：

| | 单纯形法 | 内点法 |
|---|---------|--------|
| 路径 | 沿多面体边缘走 | 从内部穿过去 |
| 适合 | 中小规模、需要热启动 | 大规模、结构稀疏 |
| 在 MIP 中的角色 | 主力——因为需要反复热启动 | 在根节点 LP 时可能更快 |

- 为什么 MIP 需要解几千几万次 LP？类比：「就像你在迷宫里反复问『如果不考虑墙，从这里到出口最短有多远？』——这个'不考虑墙'就是 LP 松弛，你需要反复问才能判断要不要继续走。」

> 🆘 你不需要手算单纯形法。理解两件事：① LP 就是线性规划求解器 ② MIP 之所以要解那么多次 LP，是因为每次分支后都要重新估计上下界。

#### 2.4 分支定界：用猜谜游戏类比讲核心算法
- 整个概念用一个「猜数游戏」贯穿：
  - 你要猜一个 1-100 之间的整数
  - 每次猜完，对方告诉你「大了」或「小了」
  - 你只需要 7 轮就能找到
  - 而如果你乱猜——最坏可能要 100 轮
  - 这就对应 MIP 中「聪明的搜索策略」和「暴力枚举」的差距——前者 7 步，后者 100 步，对于 MIP 可能是 10³ vs 10¹⁰⁰
- 映射到 MIP：
  - 分支 = 猜一个变量的值（x₁ = 0 还是 x₁ = 1？）
  - 定界 = 用 LP 松弛判断这个分支路线上「最优可能」有多好
  - 剪枝 = 确认这条路不可能比现有最好答案更好就放弃
- 分支策略：用猜谜类比
  - Strong Branching：「先试猜几个关键问题，看哪个最缩小范围，再正式猜那个」
  - Pseudo-cost Branching：「根据历史经验——之前猜这种类型的问题大概能排除多少，以后也按这个权重猜」
  - Reliability Branching（实际默认策略）：「先试几次建立经验，之后靠经验猜——Strong 和 Pseudo-cost 的混血」
- 节点选择：「你在迷宫里探索——是先把一条路探到底（DFS）？还是哪个路口看起来最有希望就走哪边（Best-First）？」
- 收敛过程：用 ASCII 画一张钳子收紧图：

```
最优值
  ▲
  │    ╲
  │     ╲___ 上界 (找到的最好解，不断降低)
  │         ╲___
  │  ─────────── 真正的全局最优(未知)
  │         ___╱
  │     ___╱
  │ ___╱     下界 (LP 松弛，不断上升)
  └─────────────────────────▶ 时间/节点数
```
- **Gap = (上界 - 下界) / |上界| × 100%**
  - 钳子收紧了 → Gap 减小 → 离最优越来越近
  - Gap = 0% → 钳子合上了 → 找到了最优解
  - Gap = 10% → 你现在的最好解最多比最优差 10%
  - 一句话：「Gap 是唯一重要的进度指标。你看日志的时候第一眼就看 H 行后面的 Gap 列，其他都是噪音。」

> 🆘 核心就三句话：分支 = 猜变量的值、定界 = 用 LP 估计潜力、剪枝 = 不可能的就跳。Gap 是进度条——不是去看节点数。

#### 2.5 割平面：不砍合法答案，只砍迷惑人的假答案
- 类比：「你在篮子里挑苹果（可行解）。LP 松弛给你的"最优解"是半个苹果（分数解）。割平面就是拿刀把那半个不存在的苹果的所在区域切掉——保证篮子里所有完整的真苹果还在。」
- 每种割只用一句话 + 一个小例子：
  - Gomory Cut：「对约束方程取整——如果 2.3x₁ + 1.7x₂ ≤ 5.5 且 x 是整数，那 x₁ + x₂ ≤ 2」
  - MIR Cut：「混合整数舍入——把分数部分凑整，产生更紧的约束」
  - Cover Cut：「背包问题里——如果三件最大的加起来超重，那这三件不能同时选」
  - Clique Cut：「A 和 B 不能同时为 1——自动检测并生成约束」
- 核心矛盾：「切太少没效果，切太多浪费算力——求解器必须在中间精准走钢丝。」
- 配一张图文描述：2D 多边形，标 LP 最优分数点 ★，然后画一条割线 ──把 ★ 划到可行域外面但多边形的所有整数点还在里面。
- 实践建议：「你不需要手动加割平面。商业求解器默认已经做得足够好。只有当你知道你的问题有特殊结构时，才考虑手动写 callback 添加。」

> 🆘 记住一句话：「割平面 = 在不动整数解的前提下把分数解所在的区域排除掉。类似切掉蛋糕的假角但不动真蛋糕。」

#### 2.6 启发式：不求最优，先求个能交差的
- 比喻：「你让实习生找西安最好的肉夹馍。他花 3 天走遍全城对比得出最优结论——精确求解。但老板说『12 点前我要看到午饭』——你让他随便去楼下买，还行就行——启发式。」
- 按「读者能记住的故事」分类（不按学术文献分类）：
  - **Rounding 类（舍入法）**：「把 LP 的小数答案硬改成整数。比如 0.7 → 1，0.3 → 0。简单粗暴，但不一定合法——舍入后可能违反约束。」
  - **Diving 类（俯冲法）**：「锁死一个变量的值，然后忽略其他变量重新求解。像打台球——瞄准一个洞后不管其他球。」
  - **修修补补类（邻域搜索）**：「先找到一个差不多的解，然后在它周围小范围搜索更好的。像你打麻将换一张牌看看胡不胡。」
  - **Sub-MIP 类（子问题精确求解）**：「把大问题里锁定大部分变量之后只剩下一个小 MIP，精确求解那个子问题。」
- 读者需要记住的不是算法名字，而是「当我跑了一天还没出可行解时，是启发式在默默救我。」
- 日志中 H 标记说明：「当你看到 H 出现在 Gurobi 日志里时，代表启发式找到了一个可行解。多个 H 重叠出现说明启发式在帮你不断改进这个解。」

> 🆘 本节唯一重要的结论：如果你跑了一天求解器都没找到一个可行解，把 `MIPFocus=1`（优先找解）。如果还是空——你的模型可能不可行，不是求解器的问题。

#### 2.7 搜索管理和终止条件：什么时候该停下来了
- 三种剪枝条件（逐条解释）：
  1. 下界 ≥ 当前上界：「这条路上的最好可能性还不如我已经找到的——放弃」
  2. LP 松弛无解：「假装整数了都没解，真的整数只会更没解——放弃」
  3. LP 松弛得到整数解：「运气好！这条路直接给出整数答案——记录然后放弃」

- Gap 收敛的典型曲线（ASCII 描述）：
```
Gap %
100 ┤╲
    ┤ ╲___
 50 ┤     ╲___
    ┤         ╲________________
  1 ┤                           ╲──── 求解器报 Gap<1%，停止
    └────────────────────────────────▶ 时间
```

- 终止条件速查表：

| 参数 | 含义 | 建议值 |
|------|------|--------|
| TimeLimit | 最多跑多久 | 业务场景设 3600s（一小时），够了 |
| MIPGap | Gap 多小就停 | 业务场景设 0.01 (1%)，研究设 0.0001 |
| NodeLimit | 最多搜多少节点 | 一般不用设，让求解器自己决定 |
| SolutionLimit | 找到多少个解就停 | 配合 MIPFocus 使用 |

- **实战铁律**：「大部分时候让 Gap 跑到 1% 就够了。你追那 0.01% 的时间够你做完下一个年度的预算了。」

> 🆘 只需要记住三个参数：TimeLimit（多久停）、MIPGap（多近就停）、MIPFocus（侧重找解还是证明最优）。

#### 2.8 实战调参：一个你不该恐惧的东西
- 先破除恐惧：「99% 的问题用默认参数就能跑出好结果。这节教你的不是调参玄学，而是当模型跑不动时该摁哪个紧急按钮。」

- 最重要的 5 个参数：

| 参数 | 干什么 | 什么时候动它 |
|------|--------|------------|
| TimeLimit | 求解器最多跑多久 | 永远设一个，别让它跑一辈子 |
| MIPGap | Gap 到多少就停 | 业务场景 1%，学术 0.01%，千万别 0 |
| MIPFocus | 1=快找可行解，2=证明最优，3=平衡 | 跑半天没解→设1；有解但 Gap 不动→设2 |
| Heuristics | 启发式强度 0-1 | 有解但不够好→加大；解太多 Gap 不动→减小 |
| Cuts | 割平面激进程度 0-3 | Gap 不动且内存够→加激进；内存不足→降级 |

- 调试决策流程图（ASCII）：

```
模型跑不动了？
  │
  ├─→ 连一个可行解都没有？
  │      └─→ MIPFocus=1, 放宽 TimeLimit
  │
  ├─→ 有解但 Gap 卡在 80% 不动？
  │      └─→ 检查数值稳定性
  │           ├─→ 系数差距 > 10⁶？→ 重新 scaling
  │           ├─→ Big-M 太大？→ 收紧 Big-M
  │           └─→ 对称性严重？→ 加对称性破坏约束
  │
  └─→ Gap 在 5% 左右龟速收敛？
         └─→ 试试 MIPFocus=2 或 MIPFocus=3
```

- **如何读懂一条 Gurobi 日志**（逐行标注式解读）：

```
Gurobi 11.0.0 log (output cut short for readability)

Nodes    |    Current Node    |     Objective Bounds      |     Work
Expl Unexpl |  Obj  Depth IntInf | Incumbent    BestBd   Gap | It/Node Time

H    0     0                  428.0000000  428.00000  0.00%     -    0s
     0     0   428.00000    0    4  428.00000  428.00000  0.00%     -    0s
H   60    57   318.0000000  318.00000  428.00000  34.6%  12.7    1s
H  140   122   289.0000000  289.00000  428.00000  48.1%  10.9    2s
H  300   244   255.0000000  255.00000  428.00000  67.8%   9.8    3s
```

逐行注释：
- `H` 开头 → 启发式找到了新可行解，Incumbent 更新
- `Expl Unexpl` → 已探索/未探索的节点数
- `Incumbent` → 当前最好的可行解的值（上界）
- `BestBd` → LP 松弛给出的下界
- `Gap` → (Incumbent - BestBd) / |Incumbent| × 100%
- 这组数据典型的收敛模式：先快速缩小 Gap（从 34.6%→48.1%...），但 Gap 在增大说明 BestBd 没怎么变（下界收紧慢），可能需要调整策略

> 🆘 「调参焦虑」很正常。记住一个真理：跑完这一段后如果问题还在，换建模方式比调参更有效。

#### 案例 1：外卖骑手的路线规划（VRP）

> 难度 ★★☆☆☆  ·  前置：读完 1.4 + 1.5  ·  求解器：HiGHS (免费)
> 预计阅读：30min  ·  预计运行：< 5min

- 业务场景：「你运营一个外卖平台，高峰期有 30 个订单、5 个骑手，怎么分配路线？」
- 建模思考三步：
  - 决定什么？哪个骑手送哪几单、顺序是什么
  - 硬约束？每单一骑手、骑手容量上限、必须回站
  - 优化什么？总行驶距离最短
- 关键建模陷阱：子环消除。用故事讲：「如果不加这个约束，求解器会偷懒——骑手 1 送 A→B→A，骑手 2 送 C→D→C，但 A 和 B 实际上是同一个餐厅的同一个骑手。求解器找了个"数学上合法但物理上荒谬"的解。」
- 代码 + 运行结果对比人工派单
- 业务价值：每天少跑 20 公里 × 5 个骑手 × 365 天 = 36,500 公里/年，按 2 元/公里 = 省 7.3 万/年（只有 30 单的规模）。扩展到 300 单：省 73 万。

#### 模型校验小节（案例 1 附赠，通用规则）

写完任何模型都必须做的三个检查：

**检查 1：极端情况测试**
把所有变量固定为 0，求解器应该报 infeasible（因为需求没满足）。如果返回 feasible——你的约束少了。

**检查 2：已知解注入**
构造一个你能心算的小实例（比如 2 个骑手、3 个订单、你知道最优路线是什么），手动把这个解作为 warm start 喂给求解器，看它能不能复现。如果能——模型大概率对了。

**检查 3：松弛检验**
去掉所有整数约束跑 LP 松弛。得到的目标值应该 ≤ 整数情况下的最小值（最小化问题）。如果 LP 值反而更大——你的模型写反了或者约束有 bug。

#### 案例 2：工厂的生产排程（Job Shop Scheduling）

> 难度 ★★★☆☆  ·  前置：读完 2.4 + 案例 1  ·  求解器：HiGHS 或 Gurobi
> 预计阅读：40min  ·  预计运行：5-15min（取决于规模）

- 业务场景：半导体厂光刻机排程——10 个晶圆批次、5 台机器、每批次在每台机器上的处理时间不同、换批次需要清洗（换型时间）
- 时间索引 vs 析取建模的取舍（一张对比表讲清楚）
- **Big-M 的 Tight Value**：「不要写 M=999999——写成 M=所有任务总时长。原因：M 太大时 LP 松弛边界极弱，B&B 得搜更多节点。把 M 收紧一个数量级可能加速 10x。」
- 核心洞察：「加了一条对称性破坏约束——如果两个批次在同一个时间能在两台机器上以同样的时间互换，锁死其中一个的顺序。这条约束让求解时间从 1 小时降到 6 分钟。」
- 业务价值：设备利用率提升 8% → 同等设备产出多 8%，年产值对应增加。

#### 案例 3：投资组合怎么选股票（MIQP 入门）

> 难度 ★★☆☆☆  ·  前置：读完 1.4 + 2.4  ·  求解器：Gurobi (需要 MIQP 能力)
> 预计阅读：25min  ·  预计运行：< 3min

- 业务场景：从 500 只股票里选 20 只建组合，既要风险分散又要收益最大化
- 为什么是离散问题？你没法买 0.3 只股票——手数必须是整数
- MIQP 快速入门：「这里的目标函数不是线性的——风险 = xᵀΣx（组合权重的二次型）。MIP + 二次目标 = MIQP。求解逻辑不变，只是 LP 松弛变成了 QP 松弛。」
- 对比三组结果：

| 方案 | 选股方式 | 年化收益 | 波动率 | Sharpe |
|------|---------|---------|--------|--------|
| 基准 | 市值前 20 | 12.3% | 18.0% | 0.57 |
| 等权分散 | 随机 20 只 | 10.1% | 14.2% | 0.55 |
| MIP 优化 | 求解器选出 | 13.8% | 14.5% | 0.78 |

- 业务价值：Sharpe 从 0.57 → 0.78，年化超额 1.5%（同类风险水平下）。管理一亿：多赚 150 万/年。

#### 案例 4：发电厂什么时候开机最省钱（Unit Commitment）

> 难度 ★★★☆☆  ·  前置：读完 2.4 + 案例 1  ·  求解器：HiGHS (免费)
> 预计阅读：35min  ·  预计运行：10-20min

- 业务场景：虚拟电厂有 10 台发电机，明天的电价每小时都不同。开机要花钱、关机要花钱、每台机发电成本不同、上下爬坡有限制。
- 为什么不用人工经验法：「人工经验表是静态的——没法预判电价波动。让你开就开让你关就关。但明天凌晨 3 点电价最低——你需要在那之前把大批机组关掉，凌晨 5 点电价回升时再开回来。这个时序决策只有优化算法能做好。」
- 全部用 HiGHS 实现（零许可成本），读者可以直接复现
- 对比结果：

| 方案 | 日运营成本 | 差异 |
|------|----------|------|
| 人工经验法 | $152,300 | — |
| MIP 优化 | $136,800 | -$15,500 (-10.2%) |

- 业务价值：日省 1.5 万 × 365 天 = 年省 560 万。一个中型电厂的规模。

#### 案例 5：电信网络怎么扩容最省钱（进阶阅读）

> 难度 ★★★★☆  ·  前置：读完 2.1-2.5 + 至少 2 个案例  ·  求解器：Gurobi + HiGHS 对比
> 预计阅读：45min  ·  预计运行：30-60min
> ⚠️ 这是进阶章节，初学者可以跳过直接做毕业项目。

- 业务场景：通信公司在 20 个城市之间铺设光纤线路，满足预估流量需求同时最小化建设成本
- 规模挑战：变量多（城市对数 × 流量类型），直接 MIP 求解可能很慢
- Benders 分解入门（只用类比，不推公式）：
  - 「把问题拆成两个角色：Master 问题（铺哪些线路）和 Sub 问题（流量怎么走）。Master 说『我铺 A-B 和 B-C』，Sub 说『这样流量走到了瓶颈』，Sub 返回一个"你下次必须铺至少 N 条线"的约束给 Master。两方博弈几轮后达成一致——这就是 Benders 分解。」
- 对比表：

| 方法 | 10 城市 | 15 城市 | 20 城市 |
|------|---------|---------|---------|
| 直接 MIP (Gurobi) | 12 秒 | 3 分钟 | > 1 小时 (Gap 15%) |
| Benders 分解 (Python 实现) | 25 秒 | 1.5 分钟 | 8 分钟 (Gap < 1%) |

- 核心洞察：「学会分解算法，你能处理的问题规模可以扩大一个数量级。但代价是你得自己写迭代循环——失去了求解器的一体化便利。」
- 业务价值：CAPEX 规划从 2.8 亿 → 2.25 亿，省 5500 万。

### 附录 A：什么时候不需要 MIP

| 你的情况 | 替代方案 | 为什么 |
|----------|---------|--------|
| 问题规模 > 10⁷ 变量 | 启发式/元启发式 | MIP 求解器在这个量级通常跑不动 |
| 不需要最优解，差不多就行 | 贪心 + 局部搜索 | 10 行代码出结果，vs 等一天跑 MIP |
| 纯约束满足（无目标函数） | CP-SAT | Google OR-Tools 的 CP-SAT 在纯离散约束问题上可能更快 |
| 线性但规模极大 + 特殊结构 | 列生成 | 用分解把问题缩小到 MIP 能处理的子问题规模 |

决策流程图（ASCII）：
```
你的问题
  │
  ├─→ 规模 < 10⁵ 变量？
  │      └─→ YES → 直接上 MIP（HiGHS 可以处理）
  │
  ├─→ 规模 10⁵-10⁶？
  │      └─→ 试试 MIP + 调参 → 2 小时内 Gap < 5%？
  │             ├─→ YES → 用 MIP
  │             └─→ NO  → 考虑启发式或分解
  │
  └─→ 规模 > 10⁶？
         └─→ 先做分解/聚合 → 然后回到 MIP
```

### 附录 B：TOP 5 初学者必踩的坑

1. **Big-M 写 999999** → 数值不稳定，LP 松弛边界极弱。始终把 M 设成问题的物理上限。
2. **忘记"每个任务只做一次"约束** → 求解器返回 cost=0 的空解。加一条 `sum(x[i,j] for j) = 1 for all i`。
3. **整数变量当连续变量用** → Gap 一直 100% 因为 LP 松弛给出分数解。检查 `vtype=GRB.INTEGER` 或 `vtype="I"`。
4. **目标函数写反** → 求解器返回你看不懂的极端答案。比如最小化成本写成了最大化的 `GRB.MAXIMIZE`。
5. **不检查 infeasibility** → 求解器说 infeasible 就放弃了，但可能是模型写错了而非真无解。用 `model.computeIIS()` (Gurobi) 或逐条注释约束查找矛盾。

### 附录 C：MIP 与机器学习的交叉点（2026 视角）

三个正在发生的融合方向：

1. **Predict-then-Optimize**：ML 预测需求 → MIP 优化决策。Uber、Amazon 物流、美团配送都是用这个范式。一句话：模型告诉你明天会有多少单，求解器告诉你该怎么派。

2. **Decision-Focused Learning**：不是先预测再优化，而是让梯度直接通过优化层反向传播——训练 ML 模型直接最小化决策损失。需要求解器的可微版本（如 CVXPY Layers、PyEPO）。

3. **LLM + MIP**：用大语言模型从自然语言描述自动生成数学模型。比如「帮我把这个生产调度问题用数学语言描述出来」→ LLM 输出 MIP 模型。2025-2026 年这个方向在快速发展，但 LLM 生成的模型目前仍需人工复核。
