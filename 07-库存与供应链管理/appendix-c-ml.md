# 附录C：库存与供应链 × ML 交叉点

机器学习和库存管理的结合是当前最热门的供应链实践方向。以下是几种典型的结合方式。

| 结合方式 | 说明 | 典型应用 |
|---------|------|---------|
| **ML 需求预测 → 库存优化** | 用 LSTM/Prophet/XGBoost 预测需求，将预测的均值和方差输入 (s,S) 模型 | 几乎所有消费品零售——沃尔玛、亚马逊、京东 |
| **强化学习补货** | 将补货策略训练为 RL policy，状态=(库存,在途,近期需求)，动作=补货量 | 大规模高维库存（千级 SKU），传统方法难以手动调参 |
| **聚类 + 差异化策略** | 用 ML 对 SKU 聚类（基于需求模式/成本/提前期），不同类用不同库存模型 | 多 SKU 库存管理——A 类用优化模型，C 类用简单规则 |
| **异常检测** | 检测供应链中的异常事件（缺货前兆、品质问题、物流延迟） | 实时供应链监控——提前预警而非事后补救 |
| **端到端决策优化** | 预测层+决策层联合训练——不再先预测后优化，而是直接优化决策 | 定价+库存联合优化——亚马逊的动态定价与补货 |

---

## 深入分析三种重要结合方式

### 1. ML 需求预测 → 库存优化（最常见）

这是目前实践中最成熟的方向。传统做法是用时间序列方法（如 Holt-Winters、ARIMA）做需求预测，然后输入到库存模型。ML 的主要贡献在于：

- **处理多维度特征**：天气、促销、节假日、竞品价格——这些特征难以纳入传统时间序列模型，但对 XGBoost 或深度学习模型来说是天然输入
- **提升预测精度**：Kaggle 上的 M5 竞赛显示，基于 ML 的方法在 SKU 级预测上比传统方法提升 10-20%

**重要提醒**：预测精度提升 ≠ 库存成本下降。预测均值的改善对库存成本影响很大，但预测方差的改善影响更大。很多 ML 模型只优化了预测均值，却没有给出可靠的概率分布——这限制了它们在库存优化中的价值。

### 2. 强化学习补货（前沿，但还不够成熟）

传统 (s,S) 策略的假设较强（需求独立同分布、成本线性），而 RL 可以处理更复杂的场景：

- 高维状态空间（多个 SKU 共享固定成本）
- 非线性成本结构（阶梯运价）
- 约束条件（供应商最小起订量、仓储上限）

**现状**：RL 补货在小规模实验中表现优异（几十个 SKU），但在大规模部署中（几千个 SKU）训练不稳定、调参困难。Google 和京东都在尝试，但公开的成功案例有限。

**谁适合尝试**：如果你有 500+ SKU、传统 (s,S) 手动调参很痛苦、并且有 ML 工程团队——可以尝试。否则先用传统方法。

### 3. 端到端决策优化（最新趋势）

传统流程：先预测需求 → 再优化决策。这个流程的问题是：预测损失函数（如 MSE）和决策损失函数（如缺货成本）不是一回事。最小化 MSE 不一定能导出最小化缺货成本的决策。

端到端方法直接优化决策：将预测模型嵌入到优化模型中，用损失就是最终的决策损失。

**适用场景**：定价+库存联合优化——都涉及决策变量（价格、补货量），目标函数是同一个（总利润）。

## 一张图判断该不该用 ML

```
      你的库存场景
     ┌─────────┐
     │ SKU 数 < 100 │ ──→ 传统模型 (EOQ/(s,S)/报童) 足够用
     └─────────┘
           ↓ SKU 数 ≥ 100
     ┌─────────────────┐
     │ 需求可预测吗？   │
     └─────────────────┘
      ↓ 是              ↓ 否
   用 ML 做预测        用 RL/启发式
   传统模型优化         不追求最优解
```

---

## 重要提醒

不要为了用 ML 而用 ML。传统库存模型（EOQ/(s,S)/报童）在 80% 的场景下已经够用。ML 的价值在于：
1. 需求预测精度提升——但这要转化为库存成本降低才算数
2. 大规模 SKU 的自动化策略——人力无法维护几千个 SKU 的手工参数
3. 无法用解析模型处理的复杂场景——但也要准备好更高的工程成本

**一个实用建议**：先在 100 个 SKU 上用传统模型做好，再逐步扩展到 ML 方案。跳过传统模型直接用 ML，就像不学加减法直接学微积分——不是不行，但你大概率会在工程细节上卡住。
