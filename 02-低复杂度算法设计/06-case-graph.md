# 案例 4：最短路径与网络流（图算法）★★★☆☆

> 「我该怎么从 A 到 B 最快？」——这不是地图导航的问题，这是计算机科学里最经典的问题之一。Dijkstra 不仅是算法，它是贪心+优先队列的完美结合。

---

## 场景

你的外卖 App 上有 10,000 个配送点。骑手要从商家出发，给沿途的 50 个顾客送餐。你需要在几毫秒内算出骑手到每个顾客的最短距离——不是「近似」，是「精确」。

暴力 BFS 在无权图上可以做到，但现实世界的地图是「有权图」——每条路有长度、耗时或拥堵指数。

**Dijkstra 算法**就是为这个场景而生的。它 1959 年被提出，到今天仍然是绝大多数最短路径问题的基础方案。

---

## 分析映射表

| 维度 | 说明 |
|------|------|
| **输入特征** | 加权图（有向/无向），边权 ≥ 0，单源起点 |
| **输出特征** | 起点到所有其他节点的最短距离 |
| **复杂度瓶颈** | 暴力 BFS 只能处理无权图；Floyd O(V³) 适合稠密小图 |
| **核心范式** | 贪心 + 优先队列（每次选「当前已知最短」的节点扩充） |
| **能否并行** | 可以（Delta-Stepping 等多级并行变体） |
| **思维难度** | ★★★☆☆ —— 理解「为什么每次选最小的就一定对」需要一点证明直觉 |

## 核心思路

### 直觉：Dijkstra 是贪心+优先队列的完美结合

理解 Dijkstra 就是理解两件事：

**1. 「我一定能保证，当前从优先队列里弹出的节点，它的最短距离已经确定了。」**

为什么？因为所有边权都是非负的。如果当前队列里最小的节点是 u（距离 d[u]），那么从任何其他路径绕到 u 的距离一定 ≥ 当前路径（因为绕路要加非负数）。所以 u 没必要再等——它的最短距离就是 d[u]。

**2. 这是贪心：每次选全局最优的局部扩展。**

和 Prim 算法（最小生成树）长得几乎一样——区别只是 Prim 累加的是到树的最小距离，Dijkstra 累加的是到源点的最短距离。

```
Dijkstra 执行过程（节点编号 0~4，起点 0）：

初始: dist = [0, ∞, ∞, ∞, ∞]
pq: [(0, 0)]

弹出 (0, 0) → 更新邻居 1 (dist=4), 邻居 2 (dist=2)
pq: [(2, 2), (4, 1)]

弹出 (2, 2) → 更新邻居 1 (dist=3, 因为 2+1<4), 邻居 3 (dist=5)
pq: [(3, 1), (4, 1)(过期), (5, 3)]

弹出 (3, 1) → 更新邻居 4 (dist=7)
pq: [(5, 3), (7, 4)]

弹出 (5, 3) → 更新邻居 4 (dist=6, 因为 5+1<7)
pq: [(6, 4), (7, 4)(过期)]

弹出 (6, 4) → 没有新更新

结果: dist = [0, 3, 2, 5, 6]
```

### 为什么 Dijkstra 不能处理负权？

**关键原因**：贪心的前提「当前最小距离已经确定」在负权边上不成立。

```
反例图（Dijkstra 会出错）：

     A ──── 2 ──── B
     │             │
     1            -5
     │             │
     └──── C ──────┘
          (路径 A→C 1, C→B -5)

起点 A:
  Dijkstra 流程:
  1. dist = [0, ∞, ∞], pq: [(0, A)]
  2. 弹出 (0, A), 更新 dist[B]=2, dist[C]=1
  3. pq: [(1, C), (2, B)]
  4. 弹出 (1, C) → 认为 dist[C]=1 已经是最优 → 更新 dist[B] = 1 + (-5) = -4
  5. 但此时 dist[B] 更新了，而从 A→B 的直接路径 dist[B]=2 已经被设为「已确定」
  6. **问题**：B 之前已经在 pq 里是 (2, B)，当 C 更新 B 为 -4 时，B 的旧记录还在队列里
  7. 弹出 (2, B) 时 Dijkstra 会认为 dist[B]=2 已经确定，但实际上最短路径是 A→C→B = -4！

  Dijkstra 结果: dist[B] = 2（错误！正确答案是 -4）
```

| 算法 | 负权 | 负环 | 单源 | 全源 | 复杂度 |
|------|------|------|------|------|--------|
| Dijkstra | ❌ | ❌ | ✅ | ❌ | O(E log V) |
| Bellman-Ford | ✅ | 可检测 | ✅ | ❌ | O(VE) |
| Floyd-Warshall | ✅ | 可检测 | ❌ | ✅ | O(V³) |
| SPFA | ✅ | 可检测 | ✅ | ❌ | O(kE) 平均 |

### 暴力 BFS vs Dijkstra vs Floyd 的规模对比表

| 算法 | 适用场景 | 顶点数 V | 边数 E | 耗时（近似） |
|------|---------|---------|-------|-----------|
| **BFS（无权图）** | 迷宫、社交网络层数 | 10⁵ | 10⁵ | 0.1s |
| **Dijkstra（堆优化）** | 地图导航、路由 | 10⁵ | 10⁶ | 0.5~1s |
| **Floyd-Warshall** | 全源最短路径（稠密小图） | 500 | 10⁵ | ~1s |
| **Floyd-Warshall** | 全源（中等规模） | 1,000 | — | ~8s |
| **Floyd-Warshall** | 全源（大图❌） | 10,000 | — | ~25 天 |

```
暴力 BFS 在无权图上 = O(V + E)
  ─ 每条边遍历一次，适合：迷宫、等权网络
  
Dijkstra 堆优化版 = O((V+E) log V)
  ─ 每条边入堆一次，每次 log V，适合：大多数现实场景
  
Floyd = O(V³)
  ─ 三层循环，适合：V ≤ 500 的全源问题
  ─ V=10000 时约 10¹² 次操作，不可接受
```

## A* 算法简介（你很可能已经用过它）

Dijkstra 有一个重要的扩展：**A* (A-Star) 算法**。如果你用过地图导航、玩过 RTS 游戏，你的每一步都在用 A*。

### A* 和 Dijkstra 的唯一区别

Dijkstra 从优先队列弹出的顺序是「从起点到当前节点的距离」。

A* 从优先队列弹出的顺序是「从起点到当前节点的距离 + 估计的剩余距离」。

```
A* 的优先级 = g(n) + h(n)

g(n) = 从起点到节点 n 的实际距离（和 Dijkstra 一样）
h(n) = 从节点 n 到终点的估计距离（启发式函数）
```

### 为什么 A* 更快？

Dijkstra 像水波纹一样向四周均匀扩散。A* 像一个知道目标方向的人——它更倾向于往终点的方向探索。

```
Dijkstra 扩散:  ● ● ● ● ● ● ● ● ● ●  （向所有方向）
                   ● ● ● ● ● ● ●
                     ● ● ● ● ●
                       ● ● ●
                         ●
A* 扩散:              ● ● ● ● ● ● （更集中在目标方向）
                           ● ● ●
                             ●
                              ●
```

| 对比维度 | Dijkstra | A* |
|---------|---------|----|
| 探索范围 | 全图均匀扩散 | 偏向目标方向 |
| 启发式 | 不需要 | 需要 h(n)，且必须满足可采纳性 |
| 保证最优 | ✅ 总是 | ✅ 当 h(n) ≤ 真实剩余距离时 |
| 典型使用 | 单源全路径 | 单源单目标 |
| 地图导航 | 不常用 | ✅ 标配 |

### h(n) 怎么选？

| 场景 | 推荐的 h(n) |
|------|-----------|
| 网格地图（四方向） | 曼哈顿距离 `|dx| + |dy|` |
| 网格地图（八方向） | 切比雪夫距离 `max(dx, dy)` |
| 欧几里得空间 | 直线距离 `√(dx² + dy²)` |
| 抽象图 | 问题特定的下界估计 |

## 关键洞察

### 洞察一：Dijkstra 是「贪心 + 优先队列」的教科书案例

很多教程把 Dijkstra 讲成「一个复杂的最短路径算法」，但它本质上只是一个贪心策略——每次选最近的点扩展——加上优先队列来高效地找到「最近的点」。

如果你理解了 Dijkstra，你就理解了：
- 为什么贪心需要**证明**（非负权边保证了贪心的正确性）
- 为什么优先队列是**工具**而不是目的（堆只是加速手段）
- 为什么算法分析不能只看复杂度不看**常数**（堆操作 log V 的常数也很小）

### 洞察二：算法选型取决于「你的图长什么样」

- 图很**小但稠密**（V=500, E=10⁵）→ Floyd 最简单，代码也最少
- 图很**大但稀疏**（V=10⁵, E=10⁶）→ Dijkstra 堆优化是唯一选择
- 图有**负权但无负环** → Bellman-Ford 或 SPFA
- 需要**全源**→ V 小用 Floyd，V 大要跑 V 次 Dijkstra

### 洞察三：算法能用不等于应该用——「够用」比「最快」更重要

Floyd 只有 5 行代码。Dijkstra 堆优化版需要 20+ 行。如果你的 V=100，Floyd 可以在 1 毫秒内完成全源计算——而你要花 10 分钟写一个 Dijkstra + 堆的代码。**选择「足够好」的算法，是工程决策的一部分。**

## 延伸与变形

| 变形 | 说明 |
|------|------|
| **A* 算法** | Dijkstra + 启发式，加速单对节点最短路径 |
| **Johnson 算法** | 先用 Bellman-Ford 处理负权，再跑 V 次 Dijkstra = 全源最短路径 O(VE log V) |
| **动态图最短路** | 边的权值随时间变化，需要动态维护最短路径树 |
| **K 短路** | 找第 1 短、第 2 短、…、第 K 短的路径（Yen 算法） |
| **最大流 / 最小割** | Ford-Fulkerson、Dinic、EK——网络流是图算法的另一个广阔天地 |
| **Delta-Stepping** | Dijkstra 的并行版本，适合 GPU/多核加速 |
