
# 附录A：什么时候不需要优化

> 全课程都在教你「怎么优化」，但有时候**不优化**才是正确的工程决策。

---

## A.1 核心原则：优化是有成本的

每一项优化都有代价：

| 成本类型 | 具体表现 |
|---------|---------|
| **代码复杂度** | 更难以理解、难以维护、新人上手慢 |
| **开发时间** | 写一个 O(n) 的解可能比 O(n²) 多花 3 倍时间 |
| **Bug 风险** | 越「精巧」的算法越容易写错边界条件 |
| **可读性** | 未来维护者（可能是你自己）能否在 5 分钟内看懂？ |

**优化 != 好代码。** 优化的目的是在「问题规模大到需要优化」时提供方案，而不是在所有场景下都用最精巧的算法。

## A.2 七种不需要优化的情况

### 情况 1：输入规模永远很小

如果数据规模确定在 n ≤ 100，O(n²) 甚至 O(n³) 都无所谓。

**真实案例**：一个每周运行一次的管理报表，处理 200 条数据，O(n²) 耗时 0.01 秒。花 3 天改成 O(n log n) 降到 0.001 秒——完全没有意义。

**判断标准**：
```
如果 暴力解耗时 × 调用频率 × 365 天 < 1 秒：
    → 不要优化
```

### 情况 2：优化使代码难以维护

**真实案例**：一段业务逻辑的核心代码，用三层循环清晰表达了「对每个用户的每个订单的每个商品做检查」。你改成用位运算 + 状态压缩，代码减少 10 行——但下一个接手的人看不懂了。

更好的做法：保持三层循环，加一个注释说「这里时间足够，不需要优化」。

### 情况 3：硬件升级更便宜

你的时间成本 > 服务器成本。

**真实案例**：优化需要 3 天（工程师成本 ≈ 5,000 元）。优化后每月节省的算力 ≈ 200 元。回本周期 = 25 个月。而直接加一台服务器（月付 500 元）立刻解决问题。

**计算方式**：
```
优化成本（开发时间 × 日薪）
vs
不优化的额外算力成本（更多 CPU / 更大内存 / 更多服务器）
```

### 情况 4：瓶颈在 IO 而不是 CPU

你的程序 90% 的时间在等数据库查询、网络请求、磁盘读写。此时优化 CPU 部分的算法完全没有意义——你应该优化的是：减少 SQL 查询次数、加缓存、加索引、批量读写。

**信号**：profile 显示 hot path 在 IO 等待中。

### 情况 5：只运行一次

一个一次性数据分析任务（数据迁移、历史数据清洗），运行一次就再也不用了。花时间优化它就像花 3 天整理一个明天就扔掉的纸箱。

### 情况 6：距离「慢」还差得很远

用户能忍受的响应时间是多少？

- Web API：< 200ms 是「很快」，< 1s 是「能接受」
- 后台任务：< 5min 是「很快」，< 30min 是「能接受」
- 批量处理：< 1h 是「很快」，< 24h 是「能接受」

如果你的算法已经在这个范围内，并且短期内不会增长——别动它。

### 情况 7：PM 没抱怨

这是最容易被忽视的信号。如果一个功能在线上跑了几个月，没人投诉慢——大概率优化它不是最高优先级的事情。

---

## A.3 优化决策检查清单

当你犹豫「要不要优化」时，问自己这 5 个问题：

```
[ ] 1. 当前解法是否已经无法满足业务需求？（响应时间、吞吐量）
[ ] 2. 输入规模是否有明确的增长预期？（数据量一个月后会翻倍？）
[ ] 3. 瓶颈真的在算法本身吗？（不是 IO？不是数据库？）
[ ] 4. 优化后代码的可维护性是否会显著下降？
[ ] 5. 优化带来的收益能否量化？（节省多少钱/多少时间？）

计分规则：
- 全部「是」→ 应该优化
- 1-2 个「是」→ 再想想，或者只优化最关键的 20%
- 0 个「是」→ 去干点别的
```

---

## A.4 结论

**本课程教你的所有技巧，是为「不得不优化」的场景准备的武器库。** 如果你连武器都不需要掏出来就能解决问题——那当然是好事。

> 就像你不会因为学会了做满汉全席，就每天花 6 小时做饭。知道怎么做，和知道什么时候该做，是两种完全不同的能力。
