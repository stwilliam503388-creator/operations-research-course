
# 附录B：TOP 5 必踩坑

> 每一个坑都是真实面试/工作场景中反复出现的。提前知道，少走一年弯路。

---

## 坑 1：DP 状态定义错了，一切都白费

**表现**：代码写完了，小数据测试通过，大数据跑出来莫名其妙的结果。调了一整天。

**根本原因**：状态定义要么遗漏了关键维度，要么包含了多余维度。

**案例**：0/1 背包，错误的状态定义是 `dp[i] = 前 i 个物品的最大价值`——完全忽略了「容量」这个维度。正确状态是 `dp[i][w] = 前 i 个物品、容量为 w 时的最大价值`。

**解毒方法**：
```
在写任何 DP 代码之前，先用手写写出：
  dp[][][] 的每一个维度代表什么？
  每个维度的范围是什么？
  最终答案在哪一个状态里？
```

画一个 3×4 的表格手动填一遍。如果填不出来 → 状态定义有问题。

## 坑 2：二分答案的判定函数写反了「可行」的条件

**表现**：二分死循环，或者永远返回 `lo` 而不是 `hi`，或者答案刚好差 1。

**根本原因**：没有想清楚单调性方向。对于「最小化最大值」型问题，小的 X 更严苛，大的 X 更宽松。对于「最大化最小值」型问题则反过来。

**案例**：木材切割，你想「每段最长能切多长」。如果 `can_cut(X)` 判定切 X 长度的段数 ≥ K 返回 True，那么 X 越大越难满足——单调递减。此时二分应该找最大的 True，模板用 `while lo < hi: mid = (lo + hi + 1)//2`。

**解毒方法**：
```
每次写二分答案前，想清楚两个问题：
1. X 越大，判定结果是 True 还是 False？
2. 我要找的是最大的 True 还是最小的 True？

然后用下面两个模板之一（别搞混）：
```

```python
# 模板 A：找最大的 True（X 越大越难满足）
while lo < hi:
    mid = (lo + hi + 1) // 2
    if can_do(mid):
        lo = mid   # 可行，试试更大的
    else:
        hi = mid - 1

# 模板 B：找最小的 True（X 越大越容易满足）
while lo < hi:
    mid = (lo + hi) // 2
    if can_do(mid):
        hi = mid   # 可行，试试更小的
    else:
        lo = mid + 1
```

## 坑 3：递归不设终止条件（或者设错了）

**表现**：`RecursionError: maximum recursion depth exceeded`。或者在递归函数里修改了全局变量却没回溯。

**根本原因**：写递归时只想着「怎么拆」没想「怎么停」。或者回溯搜索中，做了选择忘了撤销。

**案例**：N 皇后回溯：
```python
def backtrack(row):
    if row == n:                # 终止条件
        solutions.append(board[:])
        return
    for col in range(n):
        if is_safe(row, col):
            board[row] = col    # 做选择
            backtrack(row + 1)  # 递归
            board[row] = -1     # 撤销选择 ← 容易忘！
```

**解毒方法**：先写终止条件，再写递归体。把「回溯三件套」刻在脑子里：
```
1. 做选择  →  2. 递归进去  →  3. 撤销选择
```

## 坑 4：贪心以为是「对的」，其实不是

**表现**：线上环境出现了预期外的结果，PM 来问：「为什么这个订单分给了更远的骑手？」你的贪心策略在 99% 的情况下是对的——但那 1% 错了。

**根本原因**：没有严格证明贪心正确性，只凭直觉说「看起来对」。

**案例**：用「最近餐厅优先」分配骑手，但因为有超时罚款，一个远一点但 deadline 很紧的订单被忽略了——导致了巨额罚金。

**解毒方法**：
```python
# 永远做「小数据验证」
def test_greedy():
    for _ in range(1000):
        data = generate_small_random(n=5)  # n 小到可以穷举
        greedy_ans = solve_greedy(data)
        optimal_ans = solve_brute_force(data)  # 穷举所有可能
        if abs(greedy_ans - optimal_ans) > 1e-6:
            print("Greedy failed!", data)
            return
    print("All tests passed (for n=5)")
```

如果在小数据上贪心都不是最优 → 别用贪心。

## 坑 5：忽略了溢出、负数、空数组等边界条件

**表现**：`index = -1` 永远是最隐蔽的 bug。或者最大值初始化为 0，但数组全负数——结果返回 0，而正确答案是负数。

**根本原因**：只写了「正常情况」的代码，没有考虑边界。

**常见边界列表**（面试踩了 80% 的坑）：

| 场景 | 常见的坑 | 正确做法 |
|------|---------|---------|
| 空数组 | 直接访问 arr[0] | 先检查 len(arr) == 0 |
| 单元素数组 | 循环不执行或死循环 | 手动模拟 n=1 的情况 |
| 全负数 | max_ans 初始化为 0 | 初始化为 float('-inf') 或 arr[0] |
| 大整数 | Python 没事，但 C++/Java 会溢出 | 用 long/int64 |
| 索引偏移 | 二分中 lo/hi 范围搞错 | 写出来 lo=0, hi=n-1 |
| 浮点精度 | 0.1 + 0.2 != 0.3 | 比较时用 eps=1e-9 |

**解毒方法**：
```
写代码前先想三个测试用例：
1. 空输入 / n=0
2. n=1（或最小单位）
3. 最大输入（检查会不会超时/溢出）
```
