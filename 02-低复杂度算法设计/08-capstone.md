
# 🏆 毕业项目：智能外卖调度系统

> 组合至少 3 种算法范式，解决一个接近真实世界的工程问题。

---

## 问题描述

你是一家外卖平台的调度算法工程师。每天午高峰有成千上万个订单涌入，你需要设计一个调度系统，决定**每个订单分配给哪个骑手、骑手以什么路线配送**，使得总配送成本最低。

### 约束条件

| 约束 | 说明 |
|------|------|
| **骑手容量** | 每个骑手一次最多带 C 个订单（例如 C=5） |
| **时间窗口** | 每个订单有最迟送达时间 T_i，超时则罚款 P_i |
| **地理分布** | 餐厅和顾客分散在城市不同区域 |
| **动态到达** | 订单不是一次性给出的，而是实时涌入（考虑离线版本：已知全部订单） |
| **成本定义** | 总成本 = 总配送距离 + 超时罚款总额 |

### 为什么适合作为毕业项目

1. **真实感强**：组合了路径规划、资源分配、时间窗口约束多个维度
2. **范式组合自然**：不同子问题可以分配给不同的算法范式
3. **可扩展**：可以逐步加入更复杂的约束（实时动态、多目标优化）
4. **可视化**：结果可以用二维坐标画出路径

## 不同范式组合方案

### 方案一：分治 + DP + 贪心（推荐）

这是推荐的 baseline 方案，覆盖 3 种范式：

```
                      ┌───────────────────────────┐
                      │      Step 1: 区域分治       │
                      │  按地理坐标划分成 N 个区域    │
                      │  (K-Means 或网格划分)       │
                      └───────────┬───────────────┘
                                  │
                      ┌───────────▼───────────────┐
                      │      Step 2: 订单分组       │
                      │  同区域内的订单按时间窗口排序  │
                      │  → 形成「配送批次」           │
                      └───────────┬───────────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              ▼                   ▼                    ▼
    ┌─────────────────┐  ┌─────────────────┐  ┌──────────────────┐
    │ A: 区域-骑手分配  │  │ B: 批次内路径规划  │  │ C: 超时订单处理  │
    │ 贪心：把骑手分    │  │ DP (TSP) 或      │  │ 二分答案：      │
    │ 配给最近区域     │  │ 最近邻贪心       │  │ 「能接受的      │
    │ O(R·Z)         │  │ O(m²·2^m) DP    │  │ 最大超时是     │
    └─────────────────┘  └─────────────────┘  │ 多少？」       │
                                               └──────────────────┘
```

**Step 1 — 分治（区域划分）**
- 使用 K-Means（或简单的网格划分）将城市分成 K 个区域
- 每个区域内的配送问题规模缩小，可以独立求解
- **复杂度**：O(N·K·iterations)，N=订单数，K=区域数

**Step 2A — 贪心（骑手分配）**
- 每个区域统计订单数，按比例分配骑手
- 或者用「最近区域优先」贪心
- **复杂度**：O(R·Z)，R=骑手数，Z=区域数

**Step 2B — DP（批次内路径规划）**
- 每个骑手接到的订单在 C 个以内（C=5），所以批次规模很小
- 用 TSP DP（Held-Karp）：状态 dp[mask][last] 表示已访问 mask 个点、当前在 last 的最小距离
- **复杂度**：O(C²·2^C)，C=5 时恒为 O(1)
- **可降级**：如果 C 太大，回退到最近邻贪心

**Step 2C — 二分答案（超时容忍度）**
- 可选扩展：二分搜索「最大允许超时时间」，判断是否存在可行分配
- 判定函数：给定最大超时 T，能否在规定骑手数内完成所有订单？

### 方案二：纯贪心快速版

如果不要求最优，而要求秒级响应：

```
-> 订单按 deadline 排序
-> 每次取最早 deadline 的订单
-> 分配给最近且未满的骑手
-> 骑手用最近邻贪心走路线
```

**复杂度**：O(N log N + N·R) — 极快，但通常比方案一差 10-20%。

### 方案三：进阶 — 多目标 + 模拟退火

```
-> 分治划分区域（同方案一）
-> 对每个区域，跑模拟退火优化路线
-> 接受准则：有时接受更差解以跳出局部最优
-> 与 DP 方案对比，看 SA 能否在更短时间内得到接近 DP 的解
```

## 评估标准

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| **总配送距离** | 所有骑手路径距离之和 | 越低越好 |
| **超时罚金** | Σ max(0, (送达时间 - deadline)) × penalty | 越低越好 |
| **总成本** | 总距离 + 总罚金 | 核心指标 |
| **运行时间** | 算法执行时间 | ≤ 3 秒（n=200 时） |
| **骑手利用率** | 已用容量 / 总容量 | 越高越好 |
| **与上界的差距** | 方案成本 / 理论下界（所有订单直线距离之和） | 越接近 1 越好 |

### 如何测试不同方案

1. 固定随机种子，生成一组测试数据（n=50, 100, 200）
2. 分别跑方案一、方案二、方案三
3. 对比：成本 vs 时间 → 理解「trade-off」

## 扩展方向

这个项目可以无限延伸。如果你想继续深入，下面是几个方向：

### 方向 1：实时动态调度

当前假设所有订单在开始时已知。现实中订单是动态到达的——你可以：
- 把时间切成「调度窗口」（例如每 5 分钟调度一次）
- 已分配的订单不可撤回（但有调整空间）
- 这变成了一个**在线算法**问题

### 方向 2：多目标优化

现实中成本不只是距离：
- 骑手满意度（不让他们跑太远）
- 顾客等待时间方差（公平性）
- 餐厅出餐时间不确定性
- → 可以引入**帕累托前沿**分析

### 方向 3：与 ML 结合

- 用历史数据训练一个「订单完成时间预测器」
- 预测的完成时间比直线距离更准 → 用于 DP 的成本矩阵
- 见附录 C：算法优化与 ML 的交叉点

### 方向 4：改进 TSP 解法

- C=5 时 DP 没问题。如果 C=20 呢？
- 试试 **LKH（Lin-Kernighan）启发式** 或 **Christofides 算法**
- 复杂度从 O(2^C) 降到 O(C³)
