# 附录A：什么时候不需要 PDE

> PED 很强大，但不是所有问题都应该用 PDE。实际上，很大一部分工程问题用更简单的工具就能解决——用 PDE 反而走弯路。这篇附录帮你识别「什么时候收起 PDE 工具箱」。

---

## 场景一：只需要全局平均（0维建模）

这个问题听起来像「这个零件能承受多大载荷？」「这块芯片发热到多少度？」

**适用做法**：集总参数法（lumped parameter model）。假设整个系统是均匀的，用代数方程或 ODE 就能描述。

举例：估算一块铜块在空气中冷却到室温需要多长时间。
- 如果你用 PDE：划分网格、设边界条件、时间步进、后处理……折腾一整天
- 如果你用集总参数法：`T(t) = T_amb + (T_0 - T_amb)·exp(-hA/ρcV·t)`，一行公式搞定

**判断原则**：如果系统内部的温度/应力/浓度差异小于 10%（由 Biot 数 Bi = hL/k 判断），你不需要 PDE。

## 场景二：数据驱动就能解决问题

你有一堆实验数据或仿真数据，想预测某个输出。

**适用做法**：插值、回归、机器学习。

举例：你测试了 100 种不同几何的梁的刚度数据，现在想预测第 101 种几何的刚度。
- PDE 做法：每来一种新几何，重新画网格、求解、后处理
- 数据做法：用 kriging（克里金插值）或一个简单的神经网络拟合已有数据，预测时间降至毫秒级

**但是**：数据驱动在外推（extrapolation）时极其不可靠。如果你想要的几何不在已有数据的范围内，PDE 仍然是正确答案。

## 场景三：系统的响应可以事先标记（查表法）

有些物理过程太复杂或太慢，不可能实时用 PDE 求解。

**适用做法**：预计算 + 查表。

举例：引擎的 ECU 控制策略。
- ECU 需要在毫秒内决定喷油量和点火提前角
- 用 PDE 求解缸内燃烧？一秒钟都算不完一层网格
- 实际做法：预先用 PDE（或实验）算好各种工况下的最佳参数，存成 MAP 表，运行时直接查表

这其实是「离线 PDE，在线查表」的模式——PDE 仍然在使用，只是不在运行时使用。

## 场景四：设计空间可以参数化

你只想知道某个参数的变化趋势，不需要精确的空间分布。

**适用做法**：量纲分析 + 相似准则。

举例：你想知道管道的压降随流速的变化趋势。
- PDE 做法：求解 Navier-Stokes 方程，得到完整的流速和压力分布
- 量纲做法：用 Darcy-Weisbach 公式 Δp = f·(L/D)·(ρv²/2)，其中 f 由 Reynolds 数和粗糙度决定

量纲分析给不出空间分布，但它能立刻告诉你：压降跟速度的平方成正比、跟管长成正比、跟管径成反比。这种物理直觉在方案设计阶段价值巨大。

## 场景五：工程规范已经替你算好了

很多工程领域的规范标准基于保守的简化假设，已经把 PDE 解封装成了公式。

**适用做法**：直接套用规范公式。

举例：设计一根混凝土梁。
- 如果你用 PDE：需要考虑混凝土的拉伸软化、钢筋与混凝土的粘结滑移、徐变……没有半年算不明白
- 如果你用规范：Eurocode 2 或者 GB 50010 已经给出了截面承载力的计算方法，半小时算完

**但要警惕**：规范公式总是在某些假设下成立的。当你的设计超出规范适用范围时（超高性能混凝土、非常规截面形状），你必须回到 PDE。

## 决策树

```
你面临一个物理问题
│
├─ 只需要一个「大概数值」？ → 集总参数 / 量纲分析
│
├─ 有大量历史数据可用？ → 数据驱动（插值/ML）
│
├─ 需要实时响应？ → 预计算查表
│
├─ 设计在规范范围内？ → 规范公式
│
└─ 以上都不是 → 用 PDE（恭喜，你确实需要它）
```

## 核心原则

> **能不用 PDE 就不用 PDE。** PDE 是最后的手段，不是默认的工具。永远先用最简单的模型回答你最关心的问题——「这个东西究竟能不能行」「趋势对不对」。只有在简单模型无法回答、或者你需要精确空间分布时，才掏出 PDE。

工程经济学原理：一个用集总参数法花 10 分钟算出的近似答案，价值通常高于一个用 PDE 花 10 天算出的精确答案——因为前者让你在第一天就发现方向不对，而不是第 10 天。
