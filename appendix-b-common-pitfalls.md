# 附录 B：TOP 5 必踩坑

> 全文约 800 字 | 阅读时间：3 分钟

---

这里列出了运筹学初学者（以及一些老手）最常犯的五个错误。每个坑我们都见过至少十次，包括我们自己在内。

---

## 🕳️ 坑 1：忘记 Big-M 的下界

**错误示范**：
```python
# 如果建厂才能生产
produce[f] <= M * build[f]   # ❌ M 设得特别大（比如 1e9）
```

**为什么是坑**：M 太大会导致求解器的线性松弛非常弱（LP relaxation gap 很大），分支定界需要探索海量节点。

**正确做法**：M 应该尽量紧——正好等于产能上限：
```python
produce[f] <= capacity[f] * build[f]   # ✅ M = capacity[f]
```

> **口诀**：Big-M 不是 Big-Number。M 越大，解越慢。

---

## 🕳️ 坑 2：整数变量当连续变量用

**错误示范**：
```python
# 需要的卡车数量
trucks = ceil(demand / truck_capacity)  # ❌ ceil 不是线性操作
```

**为什么是坑**：`ceil` 函数使模型变成非线性，要么 MIP 无法建模，要么求解器报错。

**正确做法**：
```python
# 整数变量表示卡车数
trucks >= demand / truck_capacity        # ✅ 线性约束
trucks ∈ {0, 1, 2, ...}                  # ✅ 声明为整数
```

> **口诀**：整数就是整数，别用 ceil/floor 糊弄。

---

## 🕳️ 坑 3：量纲不统一

**错误示范**：
- 产能单位：吨/月
- 需求单位：吨/年
- 运输成本：元/吨·公里
- 建厂成本：万元

全部混在一个模型里。

**结果**：解出来的「最优方案」要么荒谬（成本为 0.001 元），要么完全不能用。

**正确做法**：在写代码之前，先写一行注释：
```python
# 所有单位：吨/年，元/吨，万元
```

> **口诀**：建模五分钟，量纲检查两分钟——这是性价比最高的两分钟。

---

## 🕳️ 坑 4：一上来就跑最优，不问是否必要

**错误示范**：拿到一个排班问题，直接上 MIP 求解器，求解器跑了 3 小时没收敛——新手说「这问题太难了」，老手说「我选错了方法」。

**正确做法**：先判断问题的「最优 vs 可行」优先级：
- 如果排班只需要一个「可行的」方案 → 用 CP 或贪心算法（秒出）
- 如果调度需要最优 → 再考虑 MIP

> **口诀**：先问「需要一个可行解还是最优解」，再选工具。

---

## 🕳️ 坑 5：忘记验证模型

**错误示范**：求解器给了最优解，Obj = 1,234,567，看着漂亮——直接出报告。

**结果**：第二天业务反馈「你的方案让我亏了 200 万」。仔细检查发现约束写错了：取消了一个约束条件，但忘记删对应的变量。

**正确做法**：永远做这三件事：
1. **已知解注入**：给一个已知最优的小问题，看求解器能不能找到
2. **极端值测试**：把需求设为 0，看是不是所有变量都变 0
3. **松弛检查**：去掉整数约束，看 LP relaxation 是否合理

> **口诀**：求解器永远在你写错的模型上返回「最优解」——它不知道你的模型对不对。

---

## 避坑检查清单

每次写完模型后，花 5 分钟检查：

- [ ] 所有 Big-M 是否设成了最小可能值？
- [ ] 所有整数变量是否声明了 `integer`？
- [ ] 目标函数和所有约束的单位一致？
- [ ] 第一步跑了 LP relaxation 检查合理性？
- [ ] 注入了一个已知解进行验证？
- [ ] 如果只需可行解，没用到 MIP？
