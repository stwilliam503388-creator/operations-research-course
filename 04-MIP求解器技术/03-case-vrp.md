# 案例1：外卖骑手的路线规划

> 难度 ★★☆☆☆  ·  前置：读完 01-background.md (1.4+1.5)  ·  求解器：HiGHS (免费)  ·  代码：code/python/case01_vrp.py  ·  预计阅读：30min  ·  预计运行：< 5min

---

## 1 场景

你运营一个外卖平台。每天中午高峰期，后台同时涌进 30 个订单——遍布在城区各个角落。你有 5 个骑手待命，每个人电动车的后备箱能装大约 7-8 份餐。

你的问题：**谁送哪几单？什么顺序送？** 目标是所有骑手行驶距离加起来最短。

直觉上，你可能会这么做：打开地图，把离得近的单分给同一个骑手，然后让骑手自己决定送单顺序。但总能找到更好的方案——比如骑手 A 本来要去小区门口，与其让骑手 B 绕路过去，不如让骑手 A 顺手多带一单。

这就是 **车辆路径规划问题 (Vehicle Routing Problem, VRP)**。它比 TSP（旅行商问题）多了一层：「先把订单分组，再对每组规划路线」。两层决策同时做，才能找到全局最优。

### 先猜一个直觉方案

如果不建模型，最自然的办法通常是：

1. 按地理位置把订单分给最近的骑手。
2. 每个骑手自己按“离当前位置最近的下一单”来送。

这个办法听起来合理，因为它每一步都在少绕路。但它的问题也在这里：**每一步近，不等于整条路线短；每个骑手局部舒服，不等于所有骑手整体最优。**

> 停一下，想一想：如果两个骑手都离某个订单差不多远，你会把它分给谁？只看距离够吗，还是还要看两个骑手后面各自会去哪？

---

## 2 模型假设与现实边界

这个案例是教学版 VRP，目的是让你看懂“订单分配 + 路线排序 + 子环消除”如何进入 MIP。为了把核心讲清楚，我们做了这些简化：

| 假设 | 教学中为什么合理 | 现实中什么时候会失效 |
|------|------------------|----------------------|
| 距离用欧氏距离近似 | 便于聚焦路线结构和 MIP 建模 | 城市道路有单行线、拥堵、红绿灯和禁行区域 |
| 所有订单一开始都已知 | 可以先讲静态 VRP | 外卖订单通常动态到达，需要滚动优化 |
| 骑手容量固定且同质 | 简化为一个容量参数 `Q` | 骑手车型、熟悉区域、服务能力可能不同 |
| 不考虑时间窗 | 先理解基本路线约束 | 真实配送强依赖承诺送达时间和取餐等待 |
| 目标只最小化总距离 | 目标函数最直观 | 业务可能更关心超时率、骑手公平性、用户体验 |

工业版通常还要加入时间窗、动态订单、交通预测、服务时长、骑手技能、区域规则和人工兜底策略。因此，本案例求出的 `optimal` 只表示“在这些教学假设下的数学最优”，不是完整外卖调度系统。

## 3 分析：从业务到变量的映射

面对一个优化问题，第一件事不是写公式，而是搞清楚三件事：你要决定什么？什么是绝对不能违反的？你要优化什么？下面用表格把 VRP 说清楚：

| 层面 | VRP 中的对应 | 说明 |
|------|-------------|------|
| **决定什么？** | 每单分给哪个骑手 + 每单的配送顺序 | 决策变量：弧选择 `x[i,j,k]` 和顺序号 `u[i,k]` |
| **硬约束？** | (1) 每单恰好被一个骑手送一次 (2) 骑手容量上限 (3) 必须从餐站出发/回到餐站 (4) 路线不能有子环 | 流量守恒 + 容量限制 + 子环消除 |
| **优化什么？** | 总行驶距离最短 | 目标函数：∑(每条走过的弧长) |

> **关键直觉**：VRP 是「先分组再排顺序」的一体化问题。如果先分组（比如按区域分配给骑手），再排顺序（每个骑手各自做 TSP），得到的是局部最优，不是全局最优。MIP 的魅力就在于这两步同时决策。

### 错误直觉：先分组再排序就够了

- **为什么听起来合理**：人脑很擅长按区域分块，看地图时也会自然把近的点圈在一起。
- **为什么会错**：订单之间的“交换”可能跨骑手发生。A 骑手顺路多带一单，B 骑手少绕一圈，总距离可能同时下降。
- **模型怎么避免**：`x[i,j,k]` 不是先固定“订单属于谁”，而是直接决定“哪个骑手走哪条弧”。分组和排序被放进同一个优化问题里一起求。

## 4 数学模型（逐行注释式）

### 4.1 集合与参数

| 符号 | 含义 | 示例值 |
|------|------|--------|
| `N` | 所有节点索引，0 = 餐站（起始点），1..n = 订单 | {0, 1, ..., 30} |
| `K` | 骑手集合 | {1, 2, 3, 4, 5} |
| `d[i,j]` | 从节点 `i` 到节点 `j` 的距离 | 欧氏距离 (km) |
| `Q` | 每个骑手的容量上限（最多同时带几单） | 8 单 |
| `M` | Big-M 常数（MTZ 子环消除用） | n+1 = 31 |

### 4.2 决策变量

```
x[i,j,k] ∈ {0, 1}    骑手 k 是否走弧 i→j（i,j 是节点号，i≠j）
u[i,k] ∈ [0, n]       骑手 k 在节点 i 的访问序号（仅 for MTZ 子环消除）
```

- `x[i,j,k]` 是主体变量——如果骑手 k 从节点 i 直接去节点 j，它就是 1，否则 0。
- `u[i,k]` 是辅助变量——它不直接出现在目标函数里，但它能防止求解器给出「数学上合法但物理上荒谬」的解。详见 §4.5。

### 4.3 目标函数

```
minimize  Σ[i∈N] Σ[j∈N] Σ[k∈K]  d[i,j] · x[i,j,k]
```

读作：所有骑手所有弧的距离之和。每走一条弧 `d[i,j]` 就累加一次，不走的弧（x=0）贡献为 0。

这是典型的**最小化总成本**目标。如果实际问题更关心「最长送达时间」，可以改成 min-max 形式（本案不展开）。

### 4.4 约束（逐条）

#### 约束 1：每单恰好被访问一次（且只被一个骑手）

```
Σ[k∈K] Σ[j∈N, j≠i]  x[i,j,k] = 1    ∀ i ∈ {1, 2, ..., n}
```

含义：对于每个订单节点 i，恰好有一条离开它的弧被选中。你不能把一单分给两个骑手，也不能忘记它。

> 通俗：每个外卖袋只会被一个骑手拎起一次。

#### 约束 2：流量守恒（进出平衡）

```
Σ[i∈N, i≠j] x[i,j,k] = Σ[h∈N, h≠j] x[j,h,k]    ∀ j ∈ N, ∀ k ∈ K
```

含义：对任意骑手 k 和任意节点 j，骑手 k 进入 j 的次数必须等于离开 j 的次数。这意味着路线是连续的——骑手不能「凭空出现在某个节点」然后消失。

> 通俗：骑手经过一个地方，就得离开这个地方。除了餐站（出发和结束都在那里）。

#### 约束 3：骑手从餐站出发限制

```
Σ[j∈{1..n}] x[0,j,k] ≤ 1    ∀ k ∈ K
```

含义：每个骑手最多从餐站出发一次。≤1 而不是 =1，允许某些骑手不被使用。

> 通俗：骑手上线才能接单，但不强制每个骑手都必须上线。

#### 约束 4：骑手容量

```
Σ[j∈{1..n}] Σ[i∈N, i≠j]  x[i,j,k] ≤ Q    ∀ k ∈ K
```

含义：骑手 k 所有「送单」的弧数 ≤ 他的外卖箱容量 Q。

> 通俗：一次最多带 Q 份餐，后备箱塞不下更多。

#### 约束 5：子环消除（MTZ 公式）— ★ 最重要的一条

```
u[i,k] - u[j,k] + M · x[i,j,k] ≤ M - 1    ∀ i,j ∈ {1..n}, i≠j, ∀ k ∈ K
```

这可能是整个模型中最「反直觉」的一条约束，我们单独讲。

---

### 4.5 子环消除的故事

假设你没有加这条约束。求解器跑完，给你这样一个结果：

```
骑手 1: 餐站 → A → B → C → 餐站  ✅ 看起来合理
骑手 2: 餐站 → D → E → 餐站      ✅ 也合理
```

但仔细检查 `x` 矩阵，你发现：

```
实际值:
  x[A, B, 1] = 1    ← 骑手1 从 A 去 B
  x[B, C, 1] = 1    ← 骑手1 从 B 去 C
  x[C, A, 1] = 1    ← 骑手1 从 C 去 A  ← 等等，这形成了一个内环！
  x[0, D, 1] = 1    ← 骑手1 从餐站去 D
  x[D, 0, 1] = 1    ← 骑手1 从 D 回餐站
```

求解器做了这样一件事：它把骑手 1 的路线拆成了两个—一个含餐站的正规路线（0→D→0），和一个悬浮的内环（A→B→C→A）。

检查所有约束：
- 约束 1（每单一次）：✓（A、B、C、D 各被访问恰好一次）
- 约束 2（流量守恒）：✓（每个节点的进出弧数相等）
- 约束 3（出发限制）：✓（骑手 1 只从餐站出发一次，骑手 2 没出发）
- 约束 4（容量）：✓

**所有约束都通过！** 但现实世界里，一个骑手怎么可能同时完成 0→D→0 和 A→B→C→A 两条独立路线？求解器「偷懒」了——它找到了一个数学上合法的解，但物理上荒谬。

#### MTZ 如何堵住这个漏洞？

MTZ (Miller-Tucker-Zemlin) 的核心思想很简单：**给路线上的每个节点一个递增的访问序号**。

```
u[i,k] = 骑手 k 在节点 i 的访问顺序号
```

- `u[depo, k] = 0` — 餐站始终是起点，序号为 0
- 如果骑手 k 从节点 i 直接去节点 j，那么必须 `u[i,k] + 1 ≤ u[j,k]`

线性化表达：

```
u[i,k] - u[j,k] + M · x[i,j,k] ≤ M - 1
```

- 当 `x[i,j,k] = 1`（确实走了 i→j）：式子变成 `u[j,k] ≥ u[i,k] + 1`，强制序号递增
- 当 `x[i,j,k] = 0`（没走这条弧）：式子变成 `u[i,k] - u[j,k] ≤ M - 1`，因为 M 足够大，自动满足，不施加任何限制

回到刚才的荒谬解：

```
骑手 1: A → B → C → A (内环)
```

MTZ 强制：`u[A,1] + 1 ≤ u[B,1] + 1 ≤ u[C,1] + 1 ≤ u[A,1]`

这意味着 `u[A,1] + 3 ≤ u[A,1]`，矛盾！所以该解被排除。

> **Big-M 取值的讲究**：不要图省事设 M = 999999。M 太大了，LP 松弛的界会变得非常松，branch-and-bound 要花大量时间在无效节点上。物理上限应该是「路径上最多经过的节点数」，即 n+1 = 31。刚刚好够用就行。

## 6 模型校验小节

写完模型不要直接拿去求解。先做三个检查——这是工业界建模的硬规矩，帮你抓出 90% 的建模错误，不用等求解器跑半天才报 infeasible。

### 检查 1：极端情况 — 所有变量固定为 0

把所有 `x[i,j,k]` 固定为 0，让求解器判断可行性。

- **期望结果**：infeasible（不可行）
- **为什么**：约束 1 要求每个订单必须有弧出发，x=0 意味着所有订单都「没被送」，直接违反约束 1
- **如果结果不是 infeasible**：说明约束 1 的右端项设错了，或者变量缺了下界

```
# 手工验证思路（代码中不包含此测试）：
# 如果对所有 i∈{1..n}: Σ x[i,*,*] = 0 < 1  →  infeasible ✓
```

### 检查 2：已知解注入 — 构造可心算的小实例

构造 2 个骑手、3 个订单的小实例，并且手动指定一个已知的最优解：

```
餐站 (0,0),  订单A (1,0),  订单B (0,1),  订单C (2,0)
骑手容量: 2 单/人

已知最优: 骑手1 送 A→B, 骑手2 送 C
  距离: 0→A→B→0 = 1+√2+1 ≈ 3.41
        0→C→0 = 2+2 = 4
  总距离 ≈ 7.41
```

- **验证**：将 `x[i,j,k]` 固定为该已知解的值，求解器应报告 feasible 且目标值 7.41
- **如果报 infeasible**：说明某条约束写错了

### 检查 3：松弛检验 — 去掉整数约束跑 LP

把 `x[i,j,k]` 从 0-1 变量放宽为连续的 `x[i,j,k] ∈ [0, 1]`，重新求解。

- **期望结果**：LP 最优值 ≤ MIP 最优值（因为 LP 的解空间更大，对最小化问题来说值只能更小或相等）
- **为什么**：整数解集合 ⊂ 连续解集合，所以最优连续解 ≤ 最优整数解
- **如果 LP 值 > MIP 值**：说明目标函数的符号搞反了，或者你跑的根本不是同一个模型

> 这三条检查做完都没问题，你就可以比较放心地求解正式实例了。

## 9 关键洞察

### 洞察 1：子环消除是整个模型的灵魂

VRP 模型的正确性几乎全靠 MTZ 约束 (或等价的 single-commodity-flow) 来保证。少了它，求解器给出 50% 距离更短的「解」——但那是因为它把一条路线拆成了两个悬浮环，现实中根本不存在。**加不加子环消除，结果的物理意义天差地别。**

### 洞察 2：Big-M 的值不是越大越好

很多教程写 `M = 999999`，这在 30 个节点的 VRP 上可能看不出区别——但如果扩展到 200 单，过大的 M 会让 LP 松弛极其松散，branch-and-bound 搜索树爆炸。**M 应该精确地设为物理上限**：路径最长经过的节点数 = n+1。这是花 30 秒思考就能省掉数小时求解时间的地方。

### 洞察 3：20%+ 的优化空间是常态，不是特例

这次模拟中 MIP 比人工排单省了 23.3%。这不是巧合——任何超过 10 个订单的配送场景，人的直觉几乎不可能穷举所有「跨骑手交换路线」的可能性。MIP 的优势恰恰在于**同时考虑所有骑手的路线，找到全局最优的分组+排序组合**。

### 如果你只记住一件事

VRP 难的不是“某个骑手怎么走最近”，而是“所有骑手如何一起分单和排序”。只看局部最近，很容易错过全局更短的路线。

## 10 业务价值

把数字换算成真金白银：

```
每天节省  19.2 km × 5 个骑手区域 × 365 天 = 35,040 km/年
按电动车成本 2 元/km (电费 + 电池折旧 + 车辆损耗)
年节省 = 35,040 × 2 ≈ 7.0 万元
```

这还只是一个区域、30 单的规模。如果一个城市有 10 个这样的区域同时运营：

```
10 区域 × 7 万 = 70 万/年
```

如果扩展到 300 单规模（更大的外卖平台标准）：

```
假设节省比例保持在 20% 左右
每天节省 ≈ 60 km × 10 区域 × 365 天 ≈ 219,000 km/年
年节省 ≈ 43.8 万元/年
```

这个数量级估算不是精确商业承诺，而是说明路线优化为什么值得投入。真实平台通常会把 VRP 思想与实时订单、交通预测、人工规则和启发式算法结合起来，形成可在线运行的调度系统。

## 11 延伸阅读

- 想换一种子环消除方法？搜索 **Single-Commodity Flow (SCF)** 公式——变量更少但约束更密，适合稀疏网络。
- 想处理时间窗（每单有最晚送达时间）？搜索 **VRPTW (VRP with Time Windows)**——在约束中加上 `arrive[i] + service[i] + travel[i,j] ≤ arrive[j] + M(1-x[i,j,k])`。
- 想理解为什么 VRP 是 NP-hard？最简单的直觉：当骑手数 = 1 时，VRP 退化为 TSP，而 TSP 本身就是 NP-hard。
- 代码中 `estimate_manual_total()` 的最近邻贪心启发式，本身就是一种构造型启发式算法——如果想深入学习启发式，可以看 Clark-Wright 节约算法 (Savings Algorithm)。
