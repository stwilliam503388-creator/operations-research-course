# 第二章 求解器技术入门

> 不推导，不炫技。只讲「它解决什么问题 → 怎么想通 → 怎么用」。

---

还记得第一章我们说的吗？MIP 是 NP-hard 问题，理论上没有万能快速解法。但现实中，求解器每天都在解几十万变量的 MIP。怎么做到的？

答案不是「数学奇迹」，而是**工程智慧**。

这一章把求解器拆成几个组件，每个组件用一个生活类比讲清楚。你不需要记住算法名，只需要记住：**求解器是一个协作团队，每个成员做自己最擅长的事。**

---

## 2.1 一张图看懂求解器架构

求解器内部不是一个黑盒，而是一个分工明确的六人小组。下面是他们在一次求解中的协作流程：

```
                    原始模型
                       │
                       ▼
              ┌─────────────────┐
              │ ① Presolve      │  ← 赛前热身：先热身拉伸，去掉多余动作
              │   模型瘦身       │
              └────────┬────────┘
                       │ 简化后的模型
                       ▼
        ┌──────────────────────────────┐
        │        分支定界主循环         │
        │                              │
        │  ┌──────────────────────┐    │
        │  │ ② LP Solver         │    │  ← 发动机：每次都要算一个 LP
        │  │   线性规划求解       │◄───┤    （MIP 的「心跳」）
        │  └──────────┬──────────┘    │
        │             │  LP 最优解     │
        │             ▼               │
        │  ┌──────────────────────┐    │
        │  │ ③ Cutting Planes    │    │  ← 削边界：切掉分数解的区域
        │  │   割平面            │    │
        │  └──────────┬──────────┘    │
        │             │               │
        │             ▼               │
        │  ┌──────────────────────┐    │
        │  │ ④ Heuristics        │    │  ← 狙击手：碰运气找整数可行解
        │  │   启发式            │    │
        │  └──────────┬──────────┘    │
        │             │               │
        │             ▼               │
        │  ┌──────────────────────┐    │
        │  │ ⑤ Branching         │    │  ← 分叉决策：变量取整还是取整？
        │  │   分支              │    │
        │  └──────────┬──────────┘    │
        │             │               │
        │             ▼               │
        │  ┌──────────────────────┐    │
        │  │ ⑥ Node Selection    │    │  ← 导航：下一步探索哪个分支？
        │  │   节点选择          │    │
        │  └──────────────────────┘    │
        │                              │
        │       循环直到 Gap 收敛       │
        └──────────────────────────────┘
                       │
                       ▼
                  最优解（或近似最优）
```

### 篮球队六人组类比

| 组件 | 篮球角色 | 干什么 | 搞不定会怎样 |
|------|----------|--------|-------------|
| ① Presolve | 赛前热身教练 | 拉伸、去掉多余装备 | 带着包袱上场，跑得慢 |
| ② LP Solver | 发动机/控球后卫 | 每回合都要运转，输出方向 | MIP 直接熄火 |
| ③ Cutting Planes | 削边界的防守 | 让「假答案」区域越来越小 | 搜索范围太大，跑断腿 |
| ④ Heuristics | 狙击手/第六人 | 偶尔一记三分，快速得分 | 半天找不到一个可行解 |
| ⑤ Branching | 分叉路口的决策者 | 「接下来猜这个变量」 | 分支爆炸，计算量失控 |
| ⑥ Node Selection | 导航员 | 决定先探索哪条路 | 在死胡同里浪费大量时间 |

> 🆘 **逃生通道：** 现在只需要理解 ② LP Solver 和 ⑤ Branching 两个概念。LP Solver 是发动机，每次都要跑；Branching 是在猜测「这个变量取整等于 0 还是 1」。其他四个组件都是**锦上添花**——没有它们求解器也能跑（只是慢）。如果你时间不够，直接跳到 2.3 和 2.4，其余小节可以在遇到实际问题时再回来查。

## 2.2 预处理：模型还没开始算，就已经被瘦身了

### 类比：编译器优化

写 C 代码时，编译器会在编译阶段把 `x * 8` 优化成 `x << 3`，把死代码删掉。你写的模型，求解器也会在正式求解前**自动做一轮优化**——这就是 Presolve（预求解）。

> **一句话本质：** Presolve 就是在计算开始之前，用逻辑推理把模型「压缩」一遍。不改变最优解，只删除冗余。

### 四种瘦身技术

**技术一：边界收紧（Bound Tightening）**

你写约束 `x + y <= 10`，且 `y >= 3`。求解器自动推出 `x <= 7`。你不需要手算——它自己会推。

```
原始：x + y ≤ 10, y ≥ 3, x ≥ 0
推理：x ≤ 10 - y ≤ 10 - 3 = 7
结果：x 的上界从 ∞ 收紧到 7
```

**技术二：冗余删除（Redundancy Removal）**

`x ≥ 0` 和 `x ≥ -5` 同时存在？后面那条删掉。`x + y ≥ 10` 和 `x + y ≥ 8`？保留 10 那一条。

**技术三：系数化简（Coefficient Reduction）**

`4x + 8y ≤ 40, x, y 为整数` → 两边同除 4（注意整数条件成立才行）：`x + 2y ≤ 10`。数值更稳定，计算更快。

**技术四：变量替换（Variable Substitution）**

如果约束里出现 `x1 + x2 = 1` 且 `x1, x2` 都是 0-1 变量，求解器直接设 `x2 = 1 - x1`，删掉一个变量。这叫「互补变量检测」。

### 冲击力数字：同一个例子走一遍

假设你丢给求解器的原始模型：
- 1,000,000 个变量
- 800,000 个约束
- 3,000,000 个非零系数

Presolve 之后：
- **100,000 个变量（减少 90%）**
- **120,000 个约束（减少 85%）**
- **600,000 个非零系数（减少 80%）**

这不是夸张，工业界真实案例（电网调度、物流网络设计）中，Presolve 减掉 80%-95% 的规模非常常见。

### 注意事项

Presolve 绝大部分时候是「纯粹的善」。但在极少数数值病态的问题上，Presolve 的代数变换可能略微改变问题的数值性质（比如把 `x / ε` 变成 `x * (1/ε)` 时放大舍入误差）。如果你怀疑 Presolve 搞出了奇怪的结果，可以在求解器参数里关掉它对比：

```
# Gurobi
model.setParam("Presolve", 0)

# HiGHS
model.setOptionValue("presolve", "off")
```

> 🆘 **逃生通道：** 记住一句话——**求解器会自动瘦身，不需要你操心。** 你只管把模型写对，别去手动化简。Presolve 做得比你手动好十倍。只在出问题时才怀疑它。

## 2.3 LP 求解器：MIP 的发动机

### 先定调：这个问题的本质，初中生都能理解

线性规划（LP，Linear Programming）是 MIP 的「连续松弛」版本——把所有「变量必须是整数」的限制暂时忘掉。

问题长这样：在几条直线围成的多边形区域里，找一个点，让某个线性函数值最大。

```
     maximize  3x + 2y
     subject to:
       x + y ≤ 4
       2x + y ≤ 5
       x, y ≥ 0
```

你拿出一张纸，画四条直线，围成一个凸多边形。最优解一定在多边形的某个顶点上。**这个结论初中几何就能理解：线性函数在凸多边形上的最大值一定在顶点取到。**

### 单纯形法：蒙眼爬山

想象你蒙着眼睛站在一个凸多面体的某个顶点上。你的目标是走到最低处（求 min）。你只能摸着棱走——每次用手试探相邻的顶点，选一个能让你下降的方向，迈一步，重复。

- **路径：** 贴着多面体的边缘，从一个顶点到另一个顶点
- **特点：** 步数可能很多（理论上最坏情况是指数级），但实际上极快，因为它利用了解的「稀疏性」
- **适合场景：** 绝大多数工业 LP，变量数远超约束数的那种稀疏问题

### 内点法：钻隧道

同样要找最低处，这次不沿边缘走了。你直接站在多面体内部，然后「钻隧道」——每一步都朝着中心的最优方向直接穿过内部，逼近最优顶点。

- **路径：** 在多面体内部穿越，最终逼近顶点
- **特点：** 迭代次数与问题规模关系稳定（多项式复杂度），但每步计算量大（要解大型线性方程组）
- **适合场景：** 大规模稠密问题，或需要极高精度的场景

### 两种方法对比

| 维度 | 单纯形法 (Simplex) | 内点法 (Barrier/IPM) |
|------|-------------------|---------------------|
| 行走路径 | 沿边缘走顶点 | 从内部穿过去 |
| 适合场景 | 稀疏问题、热启动 | 稠密问题、超大问题 |
| MIP 中的角色 | **主力**——每次分支后热启动极快 | 辅助——根节点 LP 太慢时切过去 |
| 速度感 | 像跑酷，大部分时候飞一样快 | 像钻地机，稳定的推进速度 |

现代商业求解器（Gurobi、COPT）默认会**并发跑两个算法**，谁先算完用谁的结果。这对用户完全透明。

### 为什么 MIP 要解几千几万次 LP？

这是最关键的直觉。MIP 求解器在分支定界过程中，每产生一个新的子问题（分支节点），都要解一次 LP。一个中等规模的 MIP 产生 10,000 个节点很正常——那就是 10,000 次 LP。

> **迷宫类比：** 你被困在一个迷宫（MIP）里，可以看到墙（整数约束），但不知道出口在哪。于是你不走迷宫，而是反复问一个「无视墙壁」的问题：**「假设所有墙都不存在，从当前位置到出口最短距离是多少？」** 这个距离（LP 目标值）告诉你这附近「值不值得继续探索」。距离太远？放弃这条路。距离近？在这条路深入。

翻译成 MIP 语言：「无视墙壁」= LP 松弛，「距离」= LP 最优值（作为下界/上界估计），「不值得继续」= 剪枝。

> 🆘 **逃生通道：** 不需要手算 LP。理解两个事：① LP = 一个忽略整数约束的快速估计工具；② MIP 要解很多次 LP，因为每猜一个分支方向（x=0 还是 x=1），都要重新跑 LP 看看「如果走这边，最好能走到多远」。这个反复的问和答，就是 MIP 求解的核心节奏。

## 2.4 分支定界：用猜谜游戏讲核心算法

### 猜数游戏

你心里的数字在 1 到 100 之间，我来猜。每猜一次你只说「大了」「小了」或「对了」。我会这样猜：

```
猜测 1: 50 → "大了" → 范围缩到 [1, 49]
猜测 2: 25 → "小了" → 范围缩到 [26, 49]
猜测 3: 37 → "大了" → 范围缩到 [26, 36]
猜测 4: 31 → "小了" → 范围缩到 [32, 36]
猜测 5: 34 → "大了" → 范围缩到 [32, 33]
猜测 6: 32 → "小了" → 范围缩到 [33, 33]
猜测 7: 33 → "对了！"
```

**七次，从 100 个可能中锁定唯一答案。** 这就是分支定界（Branch and Bound, B&B）的核心思想——**通过不断「猜+排除」，把搜索范围折半缩小。**

现在把这个逻辑映射到 MIP：

| 猜数游戏 | MIP 分支定界 |
|----------|------------|
| 猜一个数 | **分支 (Branch)**：选一个分数变量，强制它取整（比如 x=3.7 → 分成 x≤3 和 x≥4 两个子问题）|
| 「大了」「小了」 | **定界 (Bound)**：对每个子问题快速解一个 LP，得到「最好情况能走多远」|
| 排除范围 | **剪枝 (Prune)**：如果某分支的最好情况还不如已知可行解好，这条分支直接丢弃 |

### 钳子收紧图：对偶间隙收敛过程

求解器同时维护两个值——**上界**（已知最好整数可行解的目标值）和**下界**（所有未探索分支中最好的 LP 估计值）。它们像钳子一样从两边逼近：

```
目标值
  ▲
  │                                  
  │  上界 (Incumbent) ─────────────────────
  │        ▲ 发现可行解后更新               \
  │        │                               \
  │        │           Gap = ─────────      \
  │        ▼                               │
  │  下界 (Best Bound) ─────────────────────
  │  （所有 LP 松弛的最优值中最好的那个）
  │
  └──────────────────────────────────────────► 时间

Gap = ─────────────── × 100%
        上界

  或对称形式（最小化问题）：
       上界 - 下界
Gap = ─────────────── × 100%
         下界
```

> **「Gap 是唯一重要的进度指标——看日志第一眼就看它，不是节点数。」** 节点数再多、时间再长，只要 Gap 在下降，求解器就在干活。Gap 卡住不动了才需要介入。

### 分支策略：每次猜谁？

面对几万个变量，选哪个来分支？

| 策略 | 干什么 | 一句话 |
|------|--------|--------|
| **Strong Branching** | 对几个候选变量，每个都「先下手试算一步 LP」，选导致下界提升最大的那个 | 先试猜，挑最有效的 |
| **Pseudo-cost** | 记录历史上每个变量分支带来的目标变化，凭经验选 | 凭历史经验猜 |
| **Reliability** | 初期用 Strong Branching 积累经验，后期切换到 Pseudo-cost | 混血策略，**默认** |

### 节点选择：深度优先还是最佳优先

分支产生了一大堆待探索的节点（子问题），先处理哪个？

- **DFS（深度优先）**：一路往下钻到底，快速找到第一个可行解。类比：先找到迷宫里「任何一个出口」再说。
- **Best-First**：全局挑下界最好的节点探索，优先缩小 Gap。类比：永远走看起来最接近出口的路。
- **实际默认：** 混合策略——先 DFS 找到可行解，再 Best-First 缩小 Gap。

> 🆘 **逃生通道：** 三句话记住分支定界——① **分支就是猜**（x=0 还是 x=1？一边一条路）；② **定界就是 LP 估计**（走这条路，最好能走多远？）；③ **剪枝就是不可能就跳**（这条路最好的结果还不如我已经找到的解，不走了）。还有一个指标：**Gap 就是进度条**，看它。

## 2.5 割平面：不砍合法答案，只砍迷惑人的假答案

### 苹果篮子类比

你去超市买东西，LP 告诉你「最优方案买 3.7 个苹果」。但苹果按个卖——你不能买 3.7 个。LP 解是**分数解**，它在多面体里，但不在任何整数点上。

割平面（Cutting Planes）的思路是：在 LP 的多面体上**多画一条直线，刚好把这个 3.7 苹果的方案切出可行域外**，但保证所有整数点（3 个苹果、4 个苹果的方案）都在新可行域内。

画成图：

```
    y
    ▲
    │    ● 整数可行解（合法答案）
    │       
    │  ╱ 
    │ ╱   ★ 分数 LP 最优解 (3.7, 2.4)
    │╱  ╱
    │  ╱  割平面：y ≤ -0.5x + 4（切掉 ★ 但保留所有 ●）
    │ ╱── ─ ─ ─ ─ ─ ─
    │╱
    └──────────────────────────────► x
```

> **一句话本质：** 在保持所有整数可行解不动的前提下，把当前 LP 的分数最优解排除出去。下一次再解 LP，就不在这个假答案上了。

### 四种常见割平面

| 割的类型 | 凭啥能切 | 小例子 |
|----------|---------|--------|
| **Gomory Cut** | 从单纯形表的一行推导出来 | `x1 + 2x2 ≤ 3.5, x整数 → x1 + 2x2 ≤ 3`（小数部分直接丢掉） |
| **MIR Cut** (Mixed Integer Rounding) | 把两个约束揉在一起，利用整数条件做舍入 | `x + 0.3y ≤ 5.7, y整数 →` 推出更强的边界 |
| **Cover Cut** | 背包约束中「全选也超容量」的子集不可能同时选 | `3x₁+3x₂+3x₃+2x₄+2x₅ ≤ 8` → `x₁+x₂+x₃ ≤ 2` |
| **Clique Cut** | 从「互斥」关系推导 | `x+y ≤ 1, x,z 0-1 → x+y+z ≤ 1` 当某些二元关系成立时 |

### 核心矛盾：切多少？

- 切太少了 → LP 的还是分数解，没效果，白费一轮。
- 切太多了 → 约束矩阵越来越庞大，LP 求解本身变慢，边际效益递减。
- 求解器默认：在根节点切几轮，之后只偶尔切。工程师把这个平衡调了二十年。

> 🆘 **逃生通道：** 一句话——**割平面在不动整数解的前提下，把当前那个「骗人」的分数解切出去。** 你不用管它怎么切的，它切得动就切，切不动也不影响最终正确性。只是影响快慢。

## 2.6 启发式：不求最优，先求个能交差的

### 买肉夹馍类比

午饭时间到了，老板说「我要吃肉夹馍」。

- **学术派实习生：** 花 3 天时间，下载大众点评数据，爬取方圆 5 公里所有肉夹馍店的评论，用线性回归建模评分，最后选出最优店——但老板 12 点就要吃午饭。
- **实干派实习生：** 下楼左转，看到第一家肉夹馍店就买回来。味道 7 分，但老板 11:55 吃上了。

启发式（Heuristics）就是那个实干派。**不求最优，先找到一个可行解交差。** 在 MIP 求解中，启发式的作用是：尽快给分支定界提供一个「上界」（Incumbent），有上界才能剪枝。否则求解器一直在黑暗中摸索，所有分支都「理论上可能最优」，工作量爆炸。

### 四类启发式（不按学术分类，按你能记住的）

| 类型 | 干什么 | 类比 |
|------|--------|------|
| **Rounding 类** | 把 LP 的分数解强行「四舍五入」成整数，然后检查是否合法 | 「差一点就合法？硬凑一下试试」 |
| **Diving 类** | 沿着一个方向不断固定变量取值，一路沉到底 | 闭眼走一条路，撞墙就停 |
| **修修补补类** | 拿一个已有的整数解，局部交换/调整试图改进 | 「这个方案不错，稍微调两个变量能不能更好？」 |
| **Sub-MIP 类** | 从原问题中抽一小块出来，用精确方法求解 | 「我就优化这 10% 的变量，其他先固定」 |

### 日志中的 H 标记

在 Gurobi/COPT 的求解日志里，你会看到这样的行：

```
H   42   48                 328.00000    0.00%   582
```

这个 `H` 表示启发式（Heuristics）找到了一个值为 328 的可行解。注意看 Gap (`0.00%` 说明上界一下子被拉到下界附近了)。一个 H 行出现让 Gap 跳崖式下降，说明你之前缺的就是一个「能交差」的解。

> 🆘 **逃生通道：** 如果你跑了一整天，一个 H 都没看到，Gap 一直停在 100%——这说明求解器连一个可行解都找不到。**第一步：设 MIPFocus=1**（强制优先找可行解）。如果设了还找不到，**模型可能根本不可行**——回去检查约束是否自相矛盾。

## 2.7 搜索管理和终止条件

### 三种剪枝条件

分支定界之所以不会永远跑下去，靠的是**三种主动剪枝**：

| 条件 | 判断 | 例子 |
|------|------|------|
| **1. 不可行剪枝** | LP 无解（子问题约束矛盾） | 分支 x≥5 但前面已固定 x≤3 → 空集，直接丢弃 |
| **2. 最优性剪枝** | LP 解恰好全是整数 | 找到了这个子问题的最优解，更新 Incumbent |
| **3. 边界剪枝** | LP 最优值 ≤ 当前 Incumbent（求 min 时） | 「走这条路最好也只能走到 72，但我已经有解 70 了」→ 这条路没戏，丢弃 |

### 对偶间隙收敛曲线

```
Gap
100% ┤│
80%  ┤│╲
60%  ┤│ ╲
40%  ┤│  ╲────┐
20%  ┤│        ╲________
10%  ┤│                  ╲___
 5%  ┤│                       ╲______ ← 「长尾」：Gap 从 5% 到 0.1% 可能比从 100% 到 5% 还久
 1%  ┤│                               ╲___
 0%  └┴────────────────────────────────────────► 时间
     0   10s   1min   5min   30min   2h   一夜
```

**实战铁律：「大部分时候 Gap 到 1% 就够了。追 0.01% 的时间够做完下一年度预算了。」**

### 终止条件速查表

| 参数 | 含义 | 建议值 | 什么时候用 |
|------|------|--------|-----------|
| **TimeLimit** | 时间上限（秒） | 3600（1 小时） | 默认就设，防止跑飞 |
| **MIPGap** | 相对 Gap 阈值 | 0.01（1%） | 业务方案，1% 精度足够 |
| **NodeLimit** | 节点数上限 | 视情况 | 想快速看看松驰解就停 |
| **SolutionLimit** | 找到第 K 个解就停 | 1 或 3 | 只要几个备选方案给业务看 |
| **MIPFocus** | 搜索策略 | 1=找解 / 2=证最优 / 3=降 Gap | 核心调参参数 |

> 🆘 **逃生通道：** 你只需记住三个参数：**TimeLimit**（别让它跑没完）、**MIPGap**（1% 就够了，别追 0.001%）、**MIPFocus**（1=快给我一个解，2=我非要最优，3=Gap 卡住了帮我平衡一下）。大部分项目用这三个参数就能从「求解器在空转」变成「可控出结果」。

## 2.8 实战调参：一个你不该恐惧的东西

### 先破除恐惧

网上有很多「调参秘籍」告诉你 MIP 求解器有几百个参数。**事实是：99% 的问题，默认参数就够了。**

求解器厂商（Gurobi、COPT、HiGHS）把默认参数调了十几年，他们比你更懂怎么调。**出问题的时候，先排查模型本身，再动参数。**

### 五个最重要参数

| 参数 | 干什么 | 什么时候动它 |
|------|--------|------------|
| **MIPFocus** | 1=优先找可行解 / 2=优先证最优 / 3=平衡 | Gap 卡在 100%（没解）→设 1；Gap 降到 5% 后龟速→设 2 |
| **TimeLimit** | 限制求解时间 | 永远设，至少给个上限 |
| **MIPGap** | 相对 Gap 终止阈值 | 业务方案设 1%-5%，理论研究再设 0.01% |
| **Heuristics** | 启发式搜索时间占比 | 找不到可行解时设为 0.5（花一半时间找解） |
| **Cuts** | 割平面生成强度 | 模型「特别松」（LP 很好但整数解差很远）→设 Cuts=2 或 3 |

### 调试决策流程

```
                  ┌─────────────┐
                  │ 模型跑不动  │
                  └──────┬──────┘
                         │
                    ┌────▼────┐
                    │ 有可行解 │
                    │ 了吗？   │
                    └─┬─────┬─┘
                      │ 否  │ 是
               ┌──────▼─┐  └──────▼──────┐
               │步骤1：  │          ┌────▼────────┐
               │MIPFocus│          │ Gap 变化？   │
               │  = 1   │          └──┬──────┬───┘
               └──┬─────┘       卡100%│      │龟速
                  │             ┌────▼──┐ ┌─▼────────┐
                  │             │可能    │ │步骤3：    │
                  │             │不可行  │ │MIPFocus=2│
                  │             └───────┘ │Cuts=2    │
                  │                       └──────────┘
                  │
                  └──► 如果 MIPFocus=1
                       还找不到解，回去
                       检查模型约束是否
                       自相矛盾
```

### 如何读懂一条 Gurobi 日志

```
  Nodes | Current Node | Objective Bounds | Work
 Expl Unexpl | Obj Depth | Incumbent BestBd | Gap | It/Node Time
 ─────────────────────────────────────────────────────────────
 H    0    0                  328.00000    0.00%     -    0s
      0    0   345.20000    0  328.00000 345.200  5.24%    -   1s
   120   45   337.50000   12  328.00000 331.216  0.98%  45.2  5s
  2847  168   332.10000   18  328.87500 330.001  0.34%  52.1 30s
```

**逐列解读（类比：汽车的仪表盘）：**

| 列名 | 含义 | 类比 |
|------|------|------|
| **Expl** | 已探索节点数 | 已走过的路岔路口数 |
| **Unexpl** | 待探索节点数（队列长度） | 前方还有多少个路口等着 |
| **Obj** | 当前节点的 LP 目标值 | 当前站在这条路上，「最好能走到的」距离 |
| **Depth** | 当前节点在分支树的深度 | 分了多少次叉 |
| **Incumbent** | 当前最优整数解 | **已找到的最好方案** |
| **BestBd** | 全局下界（最小化问题） | 所有还没探索的路里，「最好的」那条的 LP 估计 |
| **Gap** | 相对 Gap | **进度条——看这个就知道还要不要等** |
| **H** | 启发式找到了新可行解 | 狙击手得分 |

看日志的流程：**先看 Gap 列 → 看 Incumbent 有没有更新 → 看 Unexpl 是涨还是跌。** 如果 Unexpl 一直在涨，求解器正在发散；如果 Unexpl 稳步下降，求解器在收敛。

> 🆘 **逃生通道：** 绝大多数慢的 MIP，**换一种建模方式比调参有效得多。** 比如你用了 Big-M 约束（一个大数 M 乘二元变量），把 Big-M 调小十个数量级，速度可能快 100 倍。调参是最后的 10%，建模方式优化才是前面的 90%。还有一句话：**默认参数跑 15 分钟，看看日志 Gap 走向，再决定要不要调参。** 不要上来就调。

## 2.9 开源 vs 商业：选哪个求解器？

### 主流求解器对比

| 求解器 | MIP 性能 | 免费 | 上手难度 | 文档质量 | Python API | 一句话定位 |
|--------|---------|------|---------|---------|-----------|-----------|
| **Gurobi** | ★★★★★ 行业最强 | 学术免费 | ★★☆ | ★★★★★ | `gurobipy` | 性能天花板，大厂标配 |
| **COPT** | ★★★★★ 接近 Gurobi | 学术免费 | ★★☆ | ★★★★ | `coptpy` | 国产之光，性价比之选 |
| **SCIP** | ★★★☆ 强但不如商业 | 完全开源 | ★★★★ | ★★★☆ | `pyscipopt` | 研究利器，支持约束编程 |
| **HiGHS** | ★★★☆ 开源最强之一 | 完全开源 | ★★★ | ★★★☆ | `highspy` | 零配置开箱即用，学生首选 |
| **CPLEX** | ★★★★★ | 商业授权 | ★★☆ | ★★★★ | `cplex` | 老牌劲旅，IBM 生态 |
| **OR-Tools** | 自带简化求解器 | 完全开源 | ★★★ | ★★★★ | 原生 Python | Google 出品，适合快速原型 |

### 一句话选购指南

- **学生 / 个人学习：→ HiGHS。** 零成本，pip install 即用，MIP 性能对中小问题（< 10 万变量）完全够用。你不需要弄学术 license。
- **公司 / 商业场景：→ Gurobi 或 COPT。** 买一个商业 license 的钱远小于一个算法工程师等求解器跑完的工资。Gurobi 生态最成熟，COPT 在某些问题上（特别是中国场景的物流调度）有惊喜。
- **学术研究 / 定制算法：→ SCIP。** 唯一支持完全定制约束处理器和分支规则的开源求解器。如果你需要把领域知识嵌入分支定界里，SCIP 是唯一选择。

### 获取方式速查

| 求解器 | 安装 |
|--------|------|
| HiGHS | `pip install highspy` |
| OR-Tools | `pip install ortools` |
| Gurobi | gurobi.com 申请学术 license → `pip install gurobipy` |
| COPT | shanshu.ai 申请 → `pip install coptpy` |
| SCIP | `pip install pyscipopt` |

> 🆘 **逃生通道：** 如果你是初学者，**先装 HiGHS 把第二章的例子跑通**。等你需要解决公司真实项目时再切换到商业求解器。刚学走路不需要买跑鞋——HiGHS 足够让你理解本章全部概念。
