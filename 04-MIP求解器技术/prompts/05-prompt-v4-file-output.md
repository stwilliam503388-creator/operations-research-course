## 核心写作约束（每个文件都遵守）

### 1. 先直觉，后公式
**永远不要上来就甩公式。** 每个新概念的结构：
> ① 先给一个生活中能摸到的类比（1-2句话）
> ② 再用白话解释数学含义（不允许堆砌符号，必须逐符号翻译成中文含义）
> ③ 最后给出正式数学表达式（此时读者已经有心理模型了）

### 2. 每个抽象概念必须配「一句话人话翻译」+ 工作场景双类比
关键概念（LP松弛、对偶、分支定界、割平面、Gap）必须给**两个类比**：一个生活类比 + 一个工作类比。

例如：
- **LP 松弛**：生活类比「假装整数变量可以是小数，先把问题变简单算一下」→ 工作类比「做预算先不算税费和零头，看大数对不对」
- **对偶**：生活类比「从卖家的角度重新看同一个交易——买家花的钱=卖家收的钱」→ 工作类比「找外包还是自己干——如果内部成本高于市场价，就该外包」
- **割平面**：生活类比「在不切掉任何合法答案的前提下，画一条线把最优但非法的分数解划出去」→ 工作类比「筛简历时划掉学历不达标的——不切掉合格的，只切掉明显不行的」
- **分支定界**：生活类比「猜谜游戏——每次猜一个变量是 0 还是 1，猜错了就回头」→ 工作类比「排查 bug 时二分定位——先测前半段，不在这半段就直接跳过」
- **Gap**：生活类比「你现在离第一名还差百分之几」→ 工作类比「你跟竞争对手的价格差——不一定非要追到 0%」

### 3. 不堆术语，建立术语地图
- 每个术语首次出现时标注英文全名，后续引用用*斜体*
- README.md 提供完整术语速查表
- 每个章节文件开头可引用「术语见 README.md」

### 4. 代码文件独立存放
- 章节 .md 文件中只放关键公式或小的代码片段（≤15行）用于讲解
- 完整可运行代码写到 `code/` 目录下独立的 .py 文件
- .py 文件顶部写清依赖（`# pip install highspy numpy pandas`）和运行方式
- .py 文件末尾必须有 `if __name__ == "__main__"` 块，包含批量测试

### 01-background.md — 第一部分：背景知识

**包含：1.1 ~ 1.5 全部内容，约 5,000 字**

#### 1.1 没有优化之前，世界是什么样子的
- 开场：你管着 50 辆货车、300 个配送点，怎么安排路线最省钱？人工排一天只能跑一半。
- 决策、约束、目标 → 做饭三要素类比（买什么菜/冰箱装不下/吃饱不超预算）
- 连续 vs 离散：「水量任意调节，但人数只能是整数——你不能派 2.7 个工人去干活」
- LP → IP → MIP 递进关系（ASCII Venn 图）
- 一句话总结：「MIP = 有些变量必须是整数的线性规划。这个"有些"让问题从简单变得极难，但也让它能描述几乎所有真实世界的决策问题。」

> 🆘 卡住了怎么办：你只需要记住三件事——决策变量、约束、目标。这就是全部的数学优化。

#### 1.2 一段八卦：MIP 的历史
- 1947 Dantzig 在 RAND 做军事后勤时"顺便"发明单纯形法（导师误以为课后作业的故事）
- 1954 旅行商问题——三人证明这个看起来简单的问题实际上是 NP-hard
- 1960 Land-Doig 分支定界——首次给出通用框架
- 1984 Karmarkar 内点法——AT&T 数学家发现完全不同的路
- 1990s-至今：CPLEX → Gurobi（核心团队出走创业）→ SCIP（开源灯塔）→ COPT（中国突围）→ HiGHS（新生代）
- ASCII 时间轴：

```
1947 ─── 1960 ───── 1984 ──────── 1990s ─────── 2010s ─── 2020s
单纯形   分支定界   内点法       CPLEX商业化   Gurobi崛起  COPT国产
                               SCIP开源                  HiGHS新生代
```
- 学这段历史的意义：MIP 不是数学玩具——背后是真实的商业竞争和工程迭代

> 🆘 这节当成背景故事看。核心记住：MIP 能走到今天是因为工业界拼命砸钱——他们真的需要这个。

#### 1.3 数学工具箱：你需要的不是证明，是直觉
- **多面体与极点**：橡皮筋套在多面体上，会崩到哪个角？
- **对偶理论**（双类比）：奶茶店定价 vs 竞争对手竞价 / 找外包还是自己干
- **NP-hardness**（双类比）：不是锁打不开——是全宇宙的计算机一起上也打不开 / 但换个锁匠可能有办法
- **松弛**（双类比）：考试先跳过难题做小题 / 先去掉最难的条件看个大概

> 🆚 对偶是本章最难的概念。如果没看懂，先记住一句话：「对偶 = 同一个最优化的两种等价描述」。

#### 1.4 五大经典模型
每种按此结构：具体场景 → 变量/约束/目标表格 → 逐行注释公式 → 典型规模

| 模型 | 场景 | 难度 | 典型规模 |
|------|------|------|---------|
| 分配问题 | 5 个骑手接 30 个订单 | ★☆☆☆☆ | 几百变量 |
| 背包问题 | 从 50 个项目里挑不超预算的组合 | ★☆☆☆☆ | 几十变量 |
| 集合覆盖 | 选最少消防站覆盖所有居民区 | ★★☆☆☆ | 几千变量 |
| 设施选址 | 全国选 10 个仓库总物流最低 | ★★☆☆☆ | 几万变量 |
| 生产排程 | 3 台机器做 20 个零件最小化时间 | ★★★☆☆ | 几千变量 |
| 网络流 | 工厂到门店怎么运最便宜 | ★★☆☆☆ | 几万到几十万 |

> 🆘 不需要全记住。选两个最贴近你工作的仔细看，其他的扫场景和变量设定。

#### 1.5 建模心法：拿到新问题，五分钟写出第一个模型
五步法：
1. **画图（最重要，2分钟）**：白板画实体和关系。标输入输出。图绘完变量位置就知道了。
2. **找决策变量**（口诀：「最后输出给领导的 Excel 表里，哪些单元格需要填？」—那就是决策变量）
3. **写约束清单（自然语言）**：列出所有「不能」—不能超重、不能重复、不能超预算
4. **写目标**（一句话）：让 X 最小/最大（通常是钱、时间、距离、利润）
5. **检查边界**：带 0 个订单？带 100 万个？去掉整数约束跑 LP？

> 🆘 如果连步骤一都画不出来——你还没理解业务。回去找业务方聊清楚。

#### 2.2 Presolve：模型还没开始算就被瘦身了

> 🆘 记住一句话：求解器比你以为的聪明得多——自动瘦身。

#### 2.3 LP求解器：MIP的发动机
- 核心问题：「在几条直线围成的区域里找一个点，让某个值最大。初中生画图都能做。」
- 单纯形法 = 爬山（沿多面体边缘走到最低点）
- 内点法 = 钻隧道（从内部直接穿过去）
- 对比表（路径/适合场景/MIP中角色）
- 为什么 MIP 需要解几千几万次 LP？迷宫类比：「反复问'如果不考虑墙，从这里到出口多远？'来判断要不要继续」

> 🆘 你不需要手算。理解：LP = 线性规划求解器，MIP 要解很多次 LP 因为每次分支后要重新估计上下界。

#### 2.4 分支定界：用猜谜游戏讲核心算法
- 猜数游戏（1-100 七轮找到）→ 映射到 MIP（分支=猜变量值 / 定界=LP估计潜力 / 剪枝=不可能就跳）
- 分支策略三种：Strong（先试猜）/ Pseudo-cost（凭历史经验）/ Reliability（混血，默认）
- 节点选择：DFS（一条路探到底）vs Best-First（最有希望的路口先走）
- Gap 收敛钳子图（ASCII）+ Gap 公式
- **唯一重要的进度指标是 Gap——不是节点数。**

> 🆘 三句话：分支=猜 / 定界=用 LP 估计 / 剪枝=不可能就跳。Gap 是进度条。

#### 2.5 割平面：不砍合法答案，只砍假答案
- 苹果篮子类比：LP 给半个苹果（分数解）→ 割平面切掉不存在苹果的区域
- 四种割各一句话 + 一个小例子：Gomory / MIR / Cover / Clique
- 核心矛盾：切太少没效果，切太多浪费算力
- 配 2D 图文描述：多边形 + LP 最优分数点 ★ + 割线 ──

> 🆘 一句话：「割平面 = 在不动整数解的前提下把分数解区域排除掉。」

#### 2.6 启发式：不求最优，先求个能交差的
- 买肉夹馍类比（实习生 3 天对比 vs 老板要 12 点前看到午饭）
- 按读者能记住的故事分类：Rounding类 / Diving类 / 修修补补类(RINS/Local Branching) / Sub-MIP类
- 日志中的 H 标记说明
- 没解时设 MIPFocus=1

> 🆘 唯一重要结论：跑了一天没可行解 → MIPFocus=1。还是空 → 模型可能不可行。

#### 2.7 搜索管理和终止条件
- 三种剪枝条件逐条解释
- Gap 收敛曲线 ASCII 图
- 终止条件速查表（TimeLimit/MIPGap/NodeLimit/SolutionLimit）
- 实战铁律：大部分时候 Gap 跑到 1% 就够了

> 🆘 只需记住三个参数：TimeLimit / MIPGap / MIPFocus。

#### 2.8 实战调参：不该恐惧的东西
- 先破除恐惧：99% 的问题默认参数就够了
- 五个最重要的参数表格（TimeLimit / MIPGap / MIPFocus / Heuristics / Cuts）
- 调试决策流程图（ASCII）：模型跑不动 → 没可行解？/ Gap 卡 80%？/ Gap 龟速？
- 如何读懂一条 Gurobi 日志（逐行标注式解读，含 H/Expl/Unexpl/Incumbent/BestBd/Gap 各列含义）

> 🆘 换建模方式比调参更有效。调参是最后手段。

#### 2.9 开源 vs 商业：选哪个求解器？
- 对比表（Gurobi/COPT/SCIP/HiGHS 的性能/免费/上手/文档/Python API）
- 一句话选购指南：学生 → HiGHS，公司 → Gurobi 或 COPT，研究定制 → SCIP

### appendix-a-when-not-mip.md — 附录A：什么时候不需要 MIP

| 情况 | 替代方案 | 为什么 |
|------|---------|--------|
| 规模 > 10⁷ | 启发式/元启发式 | MIP 跑不动 |
| 差不多就行 | 贪心 + 局部搜索 | 10 行代码出结果 vs 等一天 |
| 纯约束满足 | CP-SAT | 在纯离散约束上可能更快 |
| 线性+大规模+特殊结构 | 列生成 | 分解到 MIP 能处理的子问题 |

决策流程图（ASCII）：规模 < 10⁵ → 直接 MIP / 10⁵-10⁶ → 试试 MIP + 调参 / > 10⁶ → 先分解再 MIP

### appendix-b-common-pitfalls.md — 附录B：TOP 5 必踩坑

1. Big-M 写 999999 → 数值不稳。设成问题物理上限。
2. 忘记「每个任务只做一次」约束 → 返回空解。
3. 整数变量当连续变量 → Gap 一直 100%。
4. 目标函数写反 → 极端答案。
5. 不检查 infeasibility → 模型写错而非真无解。用 computeIIS() 或逐条注释定位。

### appendix-c-ml-intersection.md — 附录C：MIP 与 ML 交叉点

1. **Predict-then-Optimize**：ML 预测需求 → MIP 优化决策（Uber/Amazon/美团）
2. **Decision-Focused Learning**：梯度通过优化层反向传播，训练 ML 直接最小化决策损失
3. **LLM + MIP**：用大语言模型从自然语言描述自动生成数学模型（2025-26 快速发展的方向，但需人工复核）
