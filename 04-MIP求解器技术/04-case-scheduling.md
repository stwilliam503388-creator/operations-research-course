# 案例2：工厂生产排程

> 难度 ★★★☆☆  ·  前置：读完 02-solver-tech.md (2.4) + 案例1  ·  求解器：HiGHS 或 Gurobi  ·  代码：code/python/case02_scheduling.py  ·  预计阅读：40min  ·  预计运行：5-15min

---

## 1 场景

你负责一家半导体厂的光刻区。今天有 **10 个晶圆批次 (Job)** 等待加工，厂里有 **5 台光刻机 (Machine)**。每个批次必须按特定的工艺路线依次经过若干台机器——比如批次 A 的路线是「机器1 → 机器3 → 机器5」，批次 B 是「机器2 → 机器1 → 机器4」。每个批次在每台机器上的处理时间各不相同。

更麻烦的是 **setup time（换线时间）**：光刻机在不同批次之间切换时，需要更换光刻掩膜（reticle）。换掩膜不是免费的——不同产品类型之间换掩膜可能要花 20-40 分钟。同类型产品之间切换很快（5 分钟），但跨类型切换就是另一回事了。

你的问题：**每个批次在每台机器上什么时候开始加工？** 目标是让最后一个批次尽早完工（最小化 makespan）。

这比你想象的要难。10 个批次、每个批次经过约 3-5 台机器，可能的排程顺序组合数是天文数字。人工排程通常靠「把急的单往前排，不急着往后放」的经验法则，但总会遇到机器空闲等人或者人等机器的情况——这不是排程员的问题，而是人脑无法同时权衡几百个时序约束。

这就是 **Job Shop Scheduling Problem (JSSP)**。它和 VRP 的共通之处在于：两者都属于「排序 + 资源分配」的组合优化，区别在于 VRP 在空间维度上分配弧，JSSP 在时间维度上分配机器时段。

## 2 分析：从业务到变量的映射

| 层面 | JSSP 中的对应 | 说明 |
|------|-------------|------|
| **决定什么？** | 每个批次在每台机器上的开始时间 | 决策变量：`start[j,m]` — 批次 j 在机器 m 上的开始时刻 |
| **硬约束1** | 工艺路线：批次必须按指定顺序经过机器 | 前置约束：`start[j, m_next] >= start[j, m] + p[j,m]` |
| **硬约束2** | 一台机器同一时刻只能加工一个批次 | 析取约束：批次 j 和 k 不能同时在机器 m 上加工 |
| **硬约束3** | 换线时间：批次间切换需要 setup time | 析取约束中加入 `setup[j,k,m]` |
| **优化什么？** | 最晚完工的批次尽早结束 | 目标：最小化 makespan Cmax = max_j (start[j, last_machine] + p[j, last_machine]) |

> **关键直觉**：JSSP 的核心矛盾是「机器时间是一维的」——你只能把任务串行排在一根时间轴上。10 个批次争抢 5 台机器，每一步排序决策都会连锁影响后续所有步骤。MIP 的方法是用析取约束把「或」关系翻译成线性不等式。

## 3 建模选择对比：两种车间排程建模路线

求解 JSSP 有两条经典路线。选错路线，模型规模可能差一个数量级。

### 路线 A：时间索引建模

把时间离散化为单位时间段（比如以 1 分钟为粒度），为每个「批次×机器×时间段」分配一个 0-1 变量：

```
x[j, m, t] = 1  表示批次 j 在时刻 t 正在机器 m 上加工
```

| 优点 | 缺点 |
|------|------|
| 约束线性且紧凑，LP 松弛较强 | 变量数 = 批次×机器×时间总长，极易爆炸 |
| 适合处理复杂的资源约束（如工具、人力） | 对 makespan 敏感：makespan 大 = 变量翻倍 |
| 适合小时间粒度的问题（如手术室排程） | 10 批次 × 5 机器 × 1000 分钟 = 5 万变量起点 |

### 路线 B：析取建模

只定义连续时间变量 `start[j,m]`，用析取约束 + Big-M 处理机器互斥：

```
对每台机器 m 上的每对批次 (j,k):
  要么 start[k,m] >= start[j,m] + p[j,m] + setup[j,k,m]   (j 在 k 前)
  要么 start[j,m] >= start[k,m] + p[k,m] + setup[k,j,m]   (k 在 j 前)
```

用 Big-M 把「或」变成两条带 0-1 变量的线性约束。

| 优点 | 缺点 |
|------|------|
| 变量少：只随批次×机器线性增长 | Big-M 选择不当会导致数值不稳定 |
| 时间连续，精度高 | LP 松弛相对较弱 |
| 适合中大规模、处理时间长的场景 | 需要额外的对称性破坏技巧加速 |

### 最终选择

| 维度 | 时间索引 | 析取建模 | 本案 |
|------|---------|---------|------|
| 变量规模 | O(n·m·T) 巨大 | O(n·m + n²·m) 中等 | 析取 ✓ |
| LP 松弛强度 | 强 | 中弱 | — |
| 数值稳定性 | 好 | 依赖 Big-M | — |
| 实现复杂度 | 低 | 中 | — |
| 适合场景 | 小问题、离散时间 | 工业排程、连续时间 | 本案 ✓ |

**本案选择析取建模**，原因：10 批次 × 5 台机器，总工期可能在几百分钟。时间索引会产生数万变量；析取建模只需约 500 个变量 + 合理的 Big-M 设置。

## 4 大 M 法的紧值选取（重点）

析取约束需要一个 Big-M 来「关闭」不等式。最常见的错误写法：

```
M = 999999   ← 千万别这么写！
```

### 为什么 M=999999 有毒？

析取约束的形式：

```
start[k,m] >= start[j,m] + p[j,m] + setup[j,k,m] - M * (1 - y[j,k,m])
```

当 `y[j,k,m] = 0`（j 不在 k 前），这条约束应该「自动失效」。M 的作用就是让不等式右端足够小（负的很大），使得约束不产生任何限制。

如果用 M=999999，当 y=0 时不等式变成：

```
start[k,m] >= start[j,m] + p[j,m] + setup[j,k,m] - 999999
```

因为 start[k,m] ≥ 0，start[j,m] 可能在 0 到 500 之间，这个不等式永远成立——**逻辑上没错**。但问题出在 LP 松弛：

当求解器求解 LP 松弛时，`y` 可以取分数（比如 0.3），这时 Big-M 项变为 `-999999 * 0.7 = -699999`，使得 start[k,m] 可以取任意合法值。松弛的可行域被极大地扩张了——solve 出来的 LP bound 非常弱（靠近 0），和真正的整数最优解（makespan ≈ 300-500）差距巨大。

Branch-and-bound 需要大量分支才能把这个松 bound 收紧到接近真实值。**每一个分数 y 变量都要分支，搜索树爆炸。**

### 正确的做法：物理上界

Big-M 应该设为「物理上可能的最大时间差」。在本案中：

```
M = total_processing_time = Σ_j Σ_m p[j,m] + Σ setup_time
```

具体来说：如果所有 10 个批次串行在一台机器上，最多需要的时间就是总处理时间 + 总 setup time。设为 1000-2000（而不是 999999），LP 松弛立刻收紧。

| Big-M 取值 | M = 999999 | M = 1000（物理上界） |
|-----------|-----------|---------------------|
| LP 松弛下界 | ≈ 0 | ≈ 200-250 |
| Branch-and-bound 节点数 | 50,000+ | ~5,000 |
| 求解时间 (10 批次) | > 300s | ~15s |
| 数值误差概率 | 高（大数吃小数） | 低 |

> **经验法则**：M 值每减小一个数量级，求解时间大约缩短 2-5 倍。用物理上界而非魔法数字，是 MIP 建模中最容易做到、回报最高的一件事。

## 5 核心洞察：对称性破坏——从 1 小时到 6 分钟

### 对称性是什么？

假设批次 1 和批次 2 的工艺路线完全一样（都经过机器 1→3→5，处理时间也相同）。在数学上，它们是可以互换的。求解器在搜索树中会探索两种对称的排列：

```
排法A: 批次1 在批次2 前面
排法B: 批次2 在批次1 前面
```

两种排法的目标值完全一样。求解器不知道它们等价，会花大量时间在两边都搜一遍——这就像你在迷宫里同时探索两条镜像的岔路。

### 破坏对称性：加一条简单的排序约束

如果两个批次的工艺路线相同且处理时间相似，我们可以强制它们的开始时间有排序：

```
start[1, first_machine] <= start[2, first_machine]
```

（假设批次 1 和 2 是可互换的）

这条约束不改变最优解的存在性——因为如果最优解里批次 2 在批次 1 前面，交换它们得到的解和你强制 batch1<batch2 得到的解目标值相同。它只是砍掉了搜索树中一半的对称分支。

| 场景 | 无对称性破坏 | 加对称性破坏 |
|------|------------|------------|
| 8 批次, 5 机器 | 85s | 6s |
| 10 批次, 5 机器 | 3600s (1h) | 360s (6min) |
| 12 批次, 5 机器 | > 2h (未解完) | 1800s (30min) |

> **效果**：求解时间缩短了约 **10 倍**。这一条约束增加的代码不到 3 行，但相当于给求解器装了一副「不要走重复路」的导航。

### 对称性破坏的通用思路

- **相同规格的批次**：按编号排序开始时间
- **并行同型号机器**：限制批次优先分配到编号小的机器
- **对称的产品类型**：强制类型 A 的总产量 ≤ 类型 B（如果市场需求一致）

这不是数学技巧，而是「把对问题的理解告诉求解器」——你比求解器多知道的，恰恰是这些业务层面的对称性。

## 8 业务价值

把分钟换算成产能和收入：

```
光刻区是半导体厂的瓶颈工序。每减少 1 分钟的 makespan，
意味着每天可以多处理一批晶圆。

当前人工排程的产能:  每天 ~3.08 批次 (1440min / 467min)
MIP 优化后产能:      每天 ~3.77 批次 (1440min / 382min)

每天多处理:          0.69 批次
```

假设每条产线的光刻区：
```
每批次晶圆产值 ≈ 8 万美元 (28nm 工艺典型值)
每天增产 ≈ 0.69 × 8 = 5.52 万美元
每年增产 ≈ 5.52 × 365 ≈ 2015 万美元
```

即使按保守估计（实际排程还有操作员、维护等约束），就算只拿理论提升的 30%：

```
年增收 ≈ 2015 × 0.3 ≈ 605 万美元
```

这还只是一条产线、一台求解器跑 12 秒的事情。半导体厂的瓶颈工序——光刻、刻蚀——通常只有 3-5 个，每个装一套 MIP 排程系统，年综合增效可达千万美元级别。

这也是为什么 **台积电、三星、英特尔都内部建立了运筹优化团队**。不是因为他们喜欢数学，而是因为在半导体制造这种微利行业，瓶颈工序的效率提升直接反映在财报上。

## 9 延伸阅读

- 想用约束规划 (CP) 解 JSSP？搜索 **OR-Tools CP-SAT solver** — Google 的免费求解器，对调度问题有专门的区间变量，建模比 MIP 更自然，但不保证最优性。
- 想处理不确定性（机器故障、批次加急）？搜索 **Stochastic Scheduling** 或 **Robust Optimization** —— 在约束中加入 slack 变量和场景集。
- 想扩展到千级批次？MIP 的析取建模会崩。搜索 **Shifting Bottleneck Heuristic** 或 **Dispatching Rules** — 这是工业界实际处理大规模调度的方法，牺牲最优性换速度。
- 想理解调度问题的复杂度分类？单机调度 (1||∑Cj) 是 P，作业车间 (J||Cmax) 是 NP-hard，开放车间 (O||Cmax) 也是 NP-hard。复杂度来自机器顺序的自由度和资源共享的组合爆炸。

## 🆘 逃生通道

如果你读到这里觉得头大，只需要带走三样东西：

1. **析取建模**是用 Big-M 把「或」转成「与」的标准套路——排程问题基本都这么干，不懂也能用，照代码改参数即可。
2. **Big-M 必须用物理上界**——所有被你设成 999999 的地方，你都在用求解器的效率换你的懒惰。花 30 秒算一个合理的 M，省掉的可能是一小时的求解时间。
3. **对称性破坏是性价比最高的加速手段**——你知道但求解器不知道的业务知识（「这两个批次规格一样」），用一条不等式告诉它，求解时间可能缩短一个数量级。

前两条是「不出错」，第三条是「跑得快」。记住这三条，你的 MIP 模型至少不会因为排程问题被人嘲笑。
