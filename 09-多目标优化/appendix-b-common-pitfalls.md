# 附录B：TOP 5 必踩坑

> 多目标优化的坑比单目标多——因为多了一个维度。这些坑有的来自对方法本质的理解不足，有的来自常见的工程习惯。以下是新手最容易踩的 5 个坑。

---

## 坑 1：没归一化就加权求和

**症状**：权重设了等于没设，一个目标主导结果。

**例子**：
- 目标1：成本（元）——范围 0~1,000,000
- 目标2：准时率（%）——范围 0~100
- 加权求和：0.5 × 成本 + 0.5 × (1 - 准时率)
- **结果**：成本动辄几十万，准时率最多贡献 100 → 成本主导，权重白设

**更深层的问题**：这不只是量纲问题。即使两个目标量纲相同（如都是元），如果一个是百万元级、一个是千元级，同样会出现主导。归一化是对「尺度的归一」，不是对「单位的归一」。

**解决方案**：

```python
# min-max 归一化后再加权求和
cost_norm = (cost - cost_min) / (cost_max - cost_min)
service_norm = ((1 - on_time_rate) - service_min) / (service_max - service_min)
total = w1 * cost_norm + w2 * service_norm
```

**经验法则**：写完加权求和表达式后，手动代入各目标的典型值——检查是否每个目标对总和的贡献在同一量级。如果不，就归一化。

---

## 坑 2：加权求和找非凸前沿

**症状**：调了一百组权重，前沿中间还是缺一块。

**原因**：不是权重的问题，是方法的问题。加权求和等价于用一条直线（权重法线）去切可行域的目标空间——找的是直线与可行域的切点。如果帕累托前沿是凹（非凸）的，直线无法「切到」凹进去的区域。无论你调多细的权重网格，那些解永远得不到。

```
加权求和（失败）          ε-约束法（成功）
    │                     │
    │   ╱╲                │   ╱╲
    │  ╱  ╲               │  ╱  ╲
    │ ╱────╲ ← 切不到     │ ╱○ ○ ╲ ← 切到了
    │╱      ╲             │╱      ╲
    └───────              └───────
```

**解决方案**：换 ε-约束法或 NSGA-II。如果你不确定前沿是否凸——先画一次 ε-约束法得到完整前沿，然后对比加权求和的结果。如果加权求和漏掉了区域，那就是非凸。

**防坑 checklist**：
- [ ] 问题是不是二次规划？（二次凸目标→前沿通常凸）
- [ ] 目标函数有没有非线性项（如指数、三角函数、分式）？（可能有非凸区域）
- [ ] 约束中是否有互斥条件？（可能产生非凸前沿）

有任何一个为「是」，就不要只用加权求和。

---

## 坑 3：ε-约束法中 ε 选太松或太紧

**症状**：
- 太松 = 约束不起作用 → 所有 ε 值得到相同解
- 太紧 = 无可行解 → 求解器报错 or 返回空集

**原因**：ε 的合理范围 = [该目标的最小可能值, 该目标在对方最优时的值]。超出这个范围的 ε 要么无效要么无解。

**解决方案**：用各目标的单目标最优值确定 ε 的合理范围。

```python
# 1. 分别求单目标最优
f1_min = min f1(x)           # 成本最小
f2_min = min f2(x)           # 碳排放最小
f1_at_f2min = f1(x*) where x* = argmin f2  # 碳排放最小时的成本
f2_at_f1min = f2(x*) where x* = argmin f1  # 成本最小时的碳排放

# 2. ε 的合理范围
epsilons = np.linspace(f2_min, f2_at_f1min, 20)
```

**注意**：当有 3 个以上目标（ε₂, ε₃, ε₄）时，ε 的网格组合会爆炸。此时用 NSGA-II 替代 ε-约束法。

## 坑 4：NSGA-II 种群太小

**症状**：前几代就收敛到局部区域，整个前沿都覆盖不全。帕累托前沿只有寥寥几个点，且集中在某一端。

**例子**：
- 种群 = 30，决策变量 = 30 → 每个变量只有 1 个值覆盖
- 结果：所有解挤在前沿的一小段

**原因**：进化算法的搜索能力很大程度上依赖种群多样性。种群太小 → 初始覆盖不足 → 很容易收敛到局部最优 → 拥挤距离机制也救不了。

**解决方案**：
- 至少 100（经典值）
- 目标越多 → 种群越大
- 经验法则：种群 ≥ 决策变量 × 5

| 目标数 | 推荐种群 |
|-------|---------|
| 2 | 100 |
| 3 | 150~200 |
| 5+ | 300+ |

**额外提示**：如果计算资源有限，宁愿减少代数也不要减少种群。代数 100 + 种群 200 的效果通常好于代数 500 + 种群 50。

## 坑 5：把帕累托前沿上所有解直接扔给决策者

**症状**：
- 你：'这里有 100 个帕累托最优方案，您选一个吧'
- 决策者：'...'（表情从期待变成茫然然后是烦躁）

**原因**：人类决策者在面对超过 7±2 个选项时决策质量急剧下降（心理学的「选择过载」效应）。帕累托前沿上有几十上百个点——这不是帮决策者，是虐待决策者。

**解决方案**：先用聚类或 TOPSIS 降维到 5~7 个代表性解。

```python
# 用 K-means 聚类帕累托前沿
from sklearn.cluster import KMeans

front = np.array(frontier)  # (n_points, n_objectives)
kmeans = KMeans(n_clusters=5, random_state=42).fit(front)
representatives = []
for i in range(5):
    cluster_points = front[kmeans.labels_ == i]
    # 选距离中心最近的点
    center = kmeans.cluster_centers_[i]
    dists = np.linalg.norm(cluster_points - center, axis=1)
    representatives.append(cluster_points[dists.argmin()])
```

**给决策者汇报的结构**：
1. 一张帕累托前沿图（示意「我们找到了所有好方案」）
2. 一张表格，列出 5~7 个代表性方案的各目标值
3. 你的推荐（即使最终拍板权在对方——你作为分析师应该给出建议）

## Bonus 坑 6：忽略目标的方向

**看似不是坑的坑**：写代码时没注意最大化/最小化方向。

```python
# 错误示例：直接把最大化收益和最小化风险放在一起比较
A = [收益=12%, 风险=5%]
B = [收益=8%,  风险=3%]
# 没有统一成最小化方向 → dominates() 出错
```

**解决方案**：求解之前，把所有目标统一为「越小越好」——最大化取负号。
