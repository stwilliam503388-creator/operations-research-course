# 附录A：什么时候不需要多目标优化

> 多目标优化是个强大的工具，但不是所有问题都需要它。有时候，单目标优化或更简单的方法更合适——而且结果更好理解、更容易让决策者接受。

---

## 五个不需要多目标优化的场景

| 场景 | 理由 |
|------|------|
| **目标之间不冲突** | 如果 f₁ 提高时 f₂ 也自动提高——实际是单目标问题，不要人为复杂化 |
| **决策者能明确给权重且不改变** | 直接加权求和跑一次就行，不需要整个前沿 |
| **决策变量少于 3 个** | 直接穷举所有方案画雷达图给决策者看，更快、更直观 |
| **数据完全不可靠** | 多目标优化的结果精度受限于输入质量——先解决数据问题 |
| **公司文化只认「一个数」** | 多目标输出是一组方案——这不是一个数字能交差的 |

---

## 深度分析每个场景

### 场景 1：目标之间不冲突

这是最常见的误用。很多新人看到问题有多个目标就本能地套多目标优化，却忽略了检查目标之间的相关性。

**实例**：某工厂想「降低生产成本」和「提高产量」。

- 成本 = 固定成本 / 产量 + 单位变动成本
- 产量 ↑ → 单位固定成本 ↓ → 总成本 ↓
- 两个目标实际上是同一方向——提高产量同时降低了成本

**正确做法**：选其中一个作为目标（如利润最大化），另一个作为约束或干脆忽略。

**判断方法**：画出两个目标在可行域中的散点图——如果呈单调正相关，它们不冲突。

---

### 场景 2：决策者能明确给权重

有时候决策者非常清楚自己要什么：「成本对我来说占 70%，性能占 30%」。这时候不需要跑 NSGA-II 求出 100 个帕累托最优解。直接加权求和跑一次，得到一个解，汇报上去。又快又准。

**但注意**：如果决策者说「权重各 50%」但实际看方案时又挑三拣四——他其实不知道自己想要什么，那才是需要用后验方法的时候。

---

### 场景 3：决策变量少于 3 个

如果只有 2~3 个设计参数，并且每个参数的取值范围不大——枚举法比任何多目标算法都靠谱。

**实例**：设计工件的三种材料（A/B/C）和两种厚度（1mm/2mm）→ 总共 6 种组合。直接枚举，画出雷达图或柱状图给决策者看。不需要加权求和，不需要 NSGA-II。

**经验法则**：枚举量 ≤ 100 种时，枚举优先。枚举比任何算法都简单、透明、可解释。

---

### 场景 4：数据完全不可靠

多目标优化的输入是各目标函数的数值。如果这些数值本身噪声很大（比如需求预测误差 ±50%、成本估算靠拍脑袋），那么帕累托前沿的精度就是假的。

**一个常见的误区**：先跑出漂亮的帕累托前沿，再去做敏感性分析。更合理的顺序是：先提高数据质量，再做优化。数据质量差时，用多目标优化只是「精确地算错了答案」。

---

### 场景 5：公司文化只认「一个数」

有些组织（尤其是传统制造业、国企）的管理方式要求每个决策有「一个最优方案」和「一个数字」作为依据。帕累托前沿的「这里有 50 个方案，各有优劣」会被认为是不负责任。

**应对策略**：
1. 先用多目标优化找出前沿
2. 再用 TOPSIS 或层次分析法（AHP）从前沿中选出一个方案
3. 对外只汇报最终方案，内部保留前沿作为备选

---

## 决策树

```
你的问题有多个目标吗？
├── 否 → 单目标优化（结束）
└── 是 → 目标之间冲突吗？
    ├── 否 → 合并成单目标（结束）
    └── 是 → 你能提前给权重吗？
        ├── 是 → 加权求和（结束）
        └── 否 → 决策变量 < 3 吗？
            ├── 是 → 穷举枚举（结束）
            └── 否 → 数据可靠吗？
                ├── 否 → 先搞数据（结束）
                └── 是 → 公司能接受多方案吗？
                    ├── 否 → 多目标→后选一个（结束）
                    └── 是 → 你需要多目标优化！
```

## 一句话总结

> **多目标优化不是身份的象征。它的价值在于解决冲突——如果你的问题没有冲突，用最简单的方法。**
